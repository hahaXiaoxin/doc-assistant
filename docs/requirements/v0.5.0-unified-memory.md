## §0 背景与目标

### 0.1 一句话背景

**v0.4.0 真机测试暴露了一个架构级问题**：sidebar 跑在 content script 里，IndexedDB 按宿主域名（`https://bilibili.com` / `https://github.com` / `https://zhihu.com`）各自隔离——每个域名一份独立的"agent + 记忆 + 数据库"，**违背产品本意**（"所有域名共用一个 agent，上下文、数据库互通"）。配置页"记忆浏览器" Tab（扩展 origin）也因此空空如也，是同一根因的副作用。

### 0.2 从 v0.4.0 遗留问题的收敛路径

- v0.2.1 · §8 登记的"SW 与 sidebar IndexedDB 同源隔离"只影响反思 Job 一条链路，当时用"SW 只唤醒、sidebar 执行"绕过
- v0.4.0 · 需求 3"记忆浏览器 Tab"装在 **Options 页（`chrome-extension://<id>` origin）**，真机打开时空表——触发方去 bilibili 聊了 5 轮，回 Options 页想看记忆，一条没有
- 同步真机 DevTools Application 面板观察：每个访问过的宿主域名下都看到一份 `doc-assistant` IDB，**拼装起来才是完整记忆**
- 结论：这是 content-script origin 的本质问题，不是 §8 的子集；**必须把 DB 搬出宿主域名**

### 0.3 本期目标（一句话）

**把 IndexedDB 从 content-script origin 搬到扩展 origin（Offscreen Document），sidebar/options 通过消息代理读写**，从而让所有域名共用同一套记忆。顺手删掉 §8 的"SW 唤醒 sidebar"广播绕路，把反思 Job 迁到 offscreen 里直接跑。

### 0.4 核心取舍

| 问题 | 决定 |
|---|---|
| Q1 · 老数据怎么办？（各宿主域名下的 IDB） | **直接丢弃**。产品未正式发布，CHANGELOG 明示"升级后需重建记忆"即可。不做导出/迁移工具。 |
| Q2 · 为什么不用 SW 直接持 DB？ | DB 连接跟 SW 挂起/重建生命周期耦合，反思 Job 会被 Chrome 随时中断；Offscreen Document 是 MV3 官方为此类"需要长任务 + DOM/IDB"的场景提供的解 —— 常驻、不挂起、origin 统一。 |
| Q3 · 为什么不用 iframe 方案（在宿主页面嵌一个扩展域 iframe 来持 DB）？ | UX 回归成本高：selection/cross-frame focus/shadow DOM 等既有链路都要重写；Offscreen 方案对 sidebar UI 零侵入。 |
| Q4 · 老 v0.4.0 内测用户升级后会丢记忆，要不要出一次性导出工具？ | **不做**。内测用户量极少，CHANGELOG 明示即可。PM 判断：花 3h 写工具 + QA，不如把这 3h 用于加固 offscreen 架构的测试覆盖。 |

### 1.2 三个实体的职责边界

| 实体 | 进程/上下文 | 持什么 | 做什么 | 不做什么 |
|---|---|---|---|---|
| **sidebar / options（UI 层）** | content-script · 宿主 origin ／ extension options page | `RemoteMemoryStore`（代理） + UI state | 渲染、用户交互、**所有 `memory.xxx()` 调用** | 不直接 `new DexieMemoryStore()`；不持 IDB；不跑反思 Job |
| **Service Worker（路由层）** | 扩展 origin · `background/index.ts` | offscreen 的 lifecycle 所有权 | `ensureOffscreenAlive()`、转发 `MEMORY_RPC_REQUEST`、转发 `REFLECTION_TICK` | 不直接访问 IDB；不 `new DexieMemoryStore()`；不跑反思 Job |
| **Offscreen Document（数据层）** | 扩展 origin · `offscreen/index.ts` | `DexieMemoryStore`（唯一真实 DB）、`ReflectionRunner`、`ReflectionScheduler`、aux/embedding Provider 实例 | 处理 `MEMORY_RPC_REQUEST`、响应 alarm tick 跑 `runPending()` | 不暴露 UI；不监听 `chrome.alarms.onAlarm`（alarm 归 SW） |

**关键规则**：
- `DexieMemoryStore` 的 `import` 语句在仓库里**只能出现在 `apps/extension/src/offscreen/*.ts`**。sidebar bootstrap、options bootstrap、background 都**禁止**直接 `new DexieMemoryStore()`。
- 任何新增 `MemoryStore` 方法必须同步补 RPC 条目（见 §1.4），否则 RemoteMemoryStore 会在运行时报 `unknown method`。
- Offscreen 不监听 `chrome.alarms`（MV3 offscreen 文档权限不含 alarms，见 §4 风险 4）；alarm 由 SW 转发。

### 1.4 消息协议对照表（RPC）

所有 RPC 走 `chrome.runtime.sendMessage` → SW `onMessage` 路由 → offscreen（`chrome.runtime.sendMessage` 直接命中 offscreen 也可，视实现选择；**推荐 SW 做中转**以便统一日志与 offscreen 存活保障）。

**envelope 形态**：

```ts
// packages/shared/src/messaging.ts 新增
export type MessageType =
  | ...existing
  | 'MEMORY_RPC_REQUEST'
  | 'MEMORY_RPC_RESPONSE'
  | 'REFLECTION_TICK'           // SW → offscreen：alarm 触发后转发
  | 'PAGE_VISIT_ENDED';         // sidebar → offscreen：PageVisit 结束通知，登记反思任务

export interface MemoryRpcRequest {
  type: 'MEMORY_RPC_REQUEST';
  rpcId: string;                // 调用侧生成 uuid，用于匹配 response
  method: MemoryRpcMethod;
  args: unknown[];              // 位置参数，与 MemoryStore 方法签名一致
}

export interface MemoryRpcResponse {
  type: 'MEMORY_RPC_RESPONSE';
  rpcId: string;
  ok: boolean;
  result?: unknown;             // ok=true 时
  error?: { message: string; stack?: string };   // ok=false 时
}

export type MemoryRpcMethod =
  | 'remember' | 'recall' | 'deleteRecord'
  | 'listVisitSummaries' | 'listSessionTopics'
  | 'listWorkingMemories' | 'deleteWorkingMemory'
  | 'getWorkingMemory' | 'setWorkingMemory' | 'touchWorkingMemory' | 'archiveStaleWorkingMemories'
  | 'listPersonas' | 'addPersonaCandidate' | 'updatePersona'
  | 'setSessionTopic' | 'getSessionTopic'
  | 'enqueueReflection' | 'listPendingReflections' | 'updateReflection'
  | 'recordPageVisit' | 'getPageVisit'
  | 'close';
```

**每个 MemoryStore 方法对应一条 RPC**（22 条，与 `packages/memory/src/interface.ts · MemoryStore` 契约 1:1）：

| # | MemoryStore 方法 | method | request args（位置参数） | response result |
|---|---|---|---|---|
| 1 | `remember` | `remember` | `[record: MemoryRecord]` | `void` |
| 2 | `recall` | `recall` | `[query: RecallQuery]` | `MemoryRecord[]` |
| 3 | `deleteRecord` | `deleteRecord` | `[id: string]` | `void` |
| 4 | `listVisitSummaries` | `listVisitSummaries` | `[opts?: { timeRange?; limit? }]` | `MemoryRecord[]` |
| 5 | `listSessionTopics` | `listSessionTopics` | `[opts?: { limit? }]` | `SessionTopicRecord[]` |
| 6 | `listWorkingMemories` | `listWorkingMemories` | `[opts?: { limit? }]` | `WorkingMemoryRecord[]` |
| 7 | `deleteWorkingMemory` | `deleteWorkingMemory` | `[canonicalUrl: string]` | `void` |
| 8 | `getWorkingMemory` | `getWorkingMemory` | `[canonicalUrl: string]` | `WorkingMemoryRecord \| null` |
| 9 | `setWorkingMemory` | `setWorkingMemory` | `[record: WorkingMemoryRecord]` | `void` |
| 10 | `touchWorkingMemory` | `touchWorkingMemory` | `[canonicalUrl: string, at?: number]` | `void` |
| 11 | `archiveStaleWorkingMemories` | `archiveStaleWorkingMemories` | `[ttlMs: number]` | `number` |
| 12 | `listPersonas` | `listPersonas` | `[opts?: { status?; subject? }]` | `PersonaRecord[]` |

| 15 | `setSessionTopic` | `setSessionTopic` | `[record: SessionTopicRecord]` | `void` |
| 16 | `getSessionTopic` | `getSessionTopic` | `[visitId: string]` | `SessionTopicRecord \| null` |
| 17 | `enqueueReflection` | `enqueueReflection` | `[task: Omit<ReflectionTask, 'id' \| 'createdAt' \| 'attemptsCount' \| 'status'> & { id?; status? }]` | `ReflectionTask` |

| 20 | `recordPageVisit` | `recordPageVisit` | `[visit: PageVisitRecord]` | `void` |
| 21 | `getPageVisit` | `getPageVisit` | `[visitId: string]` | `PageVisitRecord \| null` |
| 22 | `close` | `close` | `[]` | `void`（offscreen 端 no-op；由 offscreen lifecycle 统管） |

**序列化边界**：
- `Float32Array`（`MemoryRecord.embedding`）**不能**经 `sendMessage` 结构化克隆直接过（会变成普通对象）。在 `RemoteMemoryStore` 序列化前转 `Array.from(embedding)`；offscreen 端还原为 `new Float32Array(arr)`。**或者**约定 embedding 永远不跨 RPC（写入端由 offscreen 自己查 embedding，调用方只传文本）—— **本期选择后者**：`MemoryRecord.embedding` 不经 RPC 序列化，由 offscreen 内部在 `remember()` 时按需补齐；调用方若在 record 上塞了 embedding，offscreen 忽略并重算（日志 warn）。
- 错误统一转成 `{ message, stack }`；`RemoteMemoryStore` 端 `throw new Error(response.error.message)`。
- RPC 超时：默认 15s（`remember` 含 embedding 计算可能较慢，但 15s 足够）；超时抛 `MemoryRpcTimeoutError`。

### 1.5 Offscreen 生命周期契约

- **创建时机**：SW 启动（`chrome.runtime.onInstalled` / `onStartup`）+ 每次收到 `MEMORY_RPC_REQUEST` 前 —— 统一封装为 `ensureOffscreenAlive()` 幂等函数（内部先 `chrome.offscreen.hasDocument()`，无则 `createDocument`）。
- **`reason`**：`'IDB_PERSISTENCE'`（enum 值 `chrome.offscreen.Reason.IDB_PERSISTENCE`）。理由：Chrome 文档明确此项用于"持久化 IDB 读写"。不用 `BLOBS`（语义不符，且审核歧义）。
- **`justification`**：`"Persist and query unified memory database across all host origins; run reflection jobs that outlive any single tab."`
- **销毁**：除非 SW 自己 `closeDocument`，否则 offscreen 不会被 Chrome 自动杀死（这是它的设计目的）。本期**不主动 close**。
- **SW 重启**（Chrome 重启扩展 / 用户点"重新加载"）：offscreen 会被一并销毁，SW 启动时重新拉起即可；offscreen 自身 bootstrap 负责重新 `new DexieMemoryStore()`（数据在 IDB 里，不会丢）。

## §4 风险清单（7 条 · 每条都要有应对）

### R1 · Offscreen 生命周期与 SW 重启

**风险**：SW 重启时 offscreen 会被一并销毁。如果仅在 `onInstalled` 里 `createDocument`，SW 唤醒 / 浏览器重启后第一条 `MEMORY_RPC_REQUEST` 会打到"不存在的 offscreen"。

**应对**：
- 统一封装 `ensureOffscreenAlive()`（幂等），**在每条 RPC 路由入口都调用一次**（已有则直接 `hasDocument()` 返回 true，开销可忽略）
- `chrome.runtime.onStartup` 也调一次，减少首个 RPC 的冷启动延迟

### R2 · 消息路由延迟累积

**风险**：SW 做中转每次 RPC 多一跳（sidebar → SW → offscreen → SW → sidebar）。高频操作如"每轮记忆召回 + WorkingMemory touch"累计可能造成肉眼可见延迟。

**应对**：
- 短期：接受（单次 RPC 量级 < 20ms，Phase2 链路每轮大约 3~5 次 RPC，可接受）
- 中期：如果真机观测到延迟，改为"sidebar 直接 sendMessage 到 offscreen"——Chrome MV3 下 offscreen 也能直接接收 `chrome.runtime.sendMessage`，SW 仅负责 `ensureOffscreenAlive`（不中转）。该优化留 v0.5.1 视数据而定，**本期不做**。
- 预埋观测：offscreen router 在每条 RPC 打 `[memory-rpc] <method> <elapsedMs>ms` 日志，便于后期评估

### R3 · `chrome.offscreen.createDocument` 的 `reason` 选择

**风险**：reason 选错可能被 Chrome 在后续版本收紧策略时关掉。

**应对**：
- 本期选 `IDB_PERSISTENCE`。理由文档清晰（"persist data in IndexedDB that other contexts can't access"），正是我们要做的事
- `justification` 写足（见 §1.5），经得起 CWS 审核 hintbar 追问
- 不用 `BLOBS`（语义是"保存用户生成的 Blob"，不符）；不用 `WORKERS`（我们没跑专用 worker）

### R4 · Offscreen 无 `chrome.alarms` 权限

**风险**：Offscreen 作为 DOM 上下文**不能**注册 `chrome.alarms.onAlarm` 监听（MV3 文档明确限制）。

**应对**：
- 采用"SW 听 alarm + 转发 REFLECTION_TICK" 设计（见 §2 PR-2）
- SW 保留 `chrome.alarms.create` + `chrome.alarms.onAlarm`，仅转发到 offscreen；不自己跑任务逻辑
- 此设计与 §8 的老方案形似（都是"SW 广播"），但**目标接收方从 sidebar 变成 offscreen**，offscreen 常驻，一定在线

### R5 · SW 冷启动延迟

**风险**：SW sleep 后 `chrome.runtime.sendMessage` 会唤醒它，但首条消息前 SW 经历冷启动（几十到几百毫秒），用户感知为"第一下点击有顿挫"。

**应对**：
- 接受（MV3 SW 冷启动是官方模型的一部分，所有扩展都会受）
- 在 SW cold start 的 top-level 代码里**立即**调 `ensureOffscreenAlive()`（不 await 也行，保证尽早开始拉起 offscreen），缩短首条 RPC 到真正处理的 gap

### R6 · 老数据用户心理承受

**风险**：已装 v0.4.0 beta 的内测用户升级后会"看起来失忆"（实际老数据还在宿主域名 IDB 里，但不再被读）。

**应对（PM 决策）**：
- **不做**一次性导出工具（内测用户量 < 10，开发成本不划算）
- CHANGELOG v0.5.0 · Breaking 段用**加粗**写清"老数据不再被读取；本次升级后请当作一次干净启动"
- 不做任何"静默迁移"尝试——跨 origin 迁移 IDB 本身就要 content-script 配合，复杂度可观

### R7 · Chrome 版本门槛

**风险**：`chrome.offscreen` 需 Chrome 109+（2022-12 发布）。早期版本用户装上扩展会 crash。

**应对**：
- `manifest.json · minimum_chrome_version: "109"`（Chrome 会直接拒绝安装，避免 crash）
- README 写清"运行环境要求"
- CWS 提交时选择对应的版本范围

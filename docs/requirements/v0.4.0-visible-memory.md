## 0. 背景与目标

v0.3.0 砍掉了 v0.1 兼容包袱后，记忆层进入"能用但不透明"状态：数据都在 IDB 黑盒里，用户既看不到、也不能按时间维度检索；Persona 一锅烩把"对 agent 的定义"和"对 user 的定义"混在一起存、混在一起注入，LLM 在第二人称语境下容易误读；话题漂移靠每 4 轮周期，无法即时响应；`host_permissions` 只覆盖千问两个域，用户自配 baseURL 的 Provider 被 CORS 拦截。

**本期目标（一句话）**：把记忆系统从黑盒变成**可见可审**的"时序自传"，并顺手把 Persona 双主体、话题漂移关键词触发、全量 host_permissions 三件小事一并结清，让 v0.4.0 成为"记忆系统对用户可见、可解释、可检索"的里程碑。

**核心取舍（已与用户拍板）**：

| 问题 | 决定 |
|---|---|
| Q1 · Persona 加 `subject` 字段，旧数据怎么办？ | **不写 Dexie upgrade 迁移脚本，不写读取兜底**；`subject: 'agent' \| 'user'` 直接**必填**。老数据若存在可以直接丢弃——产品尚未正式发布，极端情况推倒重来即可。 |
| Q2 · Chronological Index 要不要做"自动路由分支"？ | **做**。`RelevantMemorySource` 内部粗判命中"时间维查询"时，跳过向量召回，直接按时间窗注入 visit_summary 清单，让用户感知不到 tool 边界。tool 形态采用**两个 tool 分工**：`recall_memory` 保留为纯语义召回（扩参 `timeRange` / `domain` 做窗内过滤），**新增 `list_recent_visits` tool** 承担时间维列清单（`timeRange` + `domain` + `limit`）。两个 tool 语义清晰、互不耦合，避免"单 tool 双 mode"带参数分支爆炸。 |
| Q3 · "记忆" Tab 里旧的 Persona 审核占位 Card 怎么办？ | **直接删除**。不做跳转入口（审核已有 sidebar PersonaReviewBanner 承接，不造第二入口）。 |
| Q4 · 话题漂移要不要做完整版？ | **只做 0.5h 关键词触发小改进**（`shouldIdentify` 加关键词漂移检测）。完整版（相似度阈值/aux 漂移判定）留给 v0.5。 |
| Q5 · `host_permissions: <all_urls>` 要不要阻塞发版？ | **不阻塞 v0.4.0 发版**。manifest 改 `<all_urls>` + 隐私文案 + CHANGELOG breaking 说明一起打进 tag；Chrome Web Store 审核走 **tag 之后**提交流程，审核周期不挡版本节奏。 |

## 1. 最终范围（5 条需求）

> 实施顺序：**1 → 2 → 3 →（4/5 搭车做）**。1/2/3 是主菜，4/5 零散搭车。

### 需求 1 · Persona 双主体区分（`subject: 'agent' | 'user'`）

**是什么**
在 `PersonaRecord` 上新增必填字段 `subject: 'agent' | 'user'`，用于标注这条 Persona **在定义谁**：`subject='agent'` 是对 agent 自身的定义（身份、角色、性格、能力边界、行为方式），`subject='user'` 是对交流对象的定义（身份、背景、偏好）。两类定义分两类存、两类注入、两类审核。

**为什么**
Persona 的本质是**"定义"**——每条 Persona 都在回答"这是在定义谁"。v0.2.2 语义统一把 Persona 全部当"写给 Agent 的长期指令"来处理，要求模型把"用户是前端工程师"这类对用户的定义**转译**成"回答时默认用前端语境"再存进去。真机跑一段后发现：

- 模型没有 subject 标签时，第二人称语境极易混淆："你叫小瑾"到底是在定义 agent 的名字，还是在转述用户自称叫小瑾？
- 用户说"根据你的记忆我是什么人"时，Agent 没有"对用户的定义"这一视角可引用
- 反思 Job 的 prompt 要同时承担两类定义的抽取，容易串

拆成两视角后消歧义彻底解决：agent 侧定义 agent 自己（名字、性格、能力边界、行为规则都归这里），user 侧定义用户本人（背景、身份、偏好）。两类定义协同工作——agent 知道用户是前端，就用前端术语举例；知道用户偏好简洁，就少讲废话。user 侧的定义直接影响 agent 如何表达。

**实施方案**

1. **Schema**（`packages/memory/src/interface.ts`）
   - `PersonaRecord` 新增必填字段：
     ```ts
     /** 主体视角（v0.4.0 新增必填） */
     subject: 'agent' | 'user';
     ```
   - 文档注释同步更新：
     - `subject='agent'`：**对 agent 的定义**——身份、角色、性格、能力边界、行为方式。例如"你叫小瑾，是文档阅读助手"、"你回答要简洁"、"你的语气偏技术向"
     - `subject='user'`：**对 user 的定义**——身份、背景、偏好。例如"用户是前端工程师"、"用户偏好 TypeScript"、"用户母语是中文"
   - **不写 Dexie upgrade 迁移脚本**（按 Q1 决策）；Dexie `version()` 号不升。老数据里缺 `subject` 字段的 PersonaRecord 读出时会在 TS 层 undefined，**不做兜底**，由 developer 在本地清一下 `persona` 表即可（产品未发布，无外部用户）。
   - `listPersonas(opts?)` 的 opts 新增 `subject?: 'agent' | 'user'` 过滤。

2. **写入路径**
   - **Tool · `remember_persona`**（`packages/tools/src/definitions/remember-persona.ts`）
     - `parametersJsonSchema` 加 `subject: { type: 'string', enum: ['agent', 'user'] }`，**required** 里加 `subject`
     - description 核心判断标准：**这条信息在定义谁？**
       - `subject='agent'` → 在定义 **agent 是谁**：身份、角色、名字、性格、能力边界、行为方式、风格与术语约定
         - 例：
           - "你叫小瑾，是我的文档阅读助手" → agent（定义身份）
           - "你回答要简洁，少讲废话" → agent（定义行为方式）
           - "遇到代码先讲结论再贴示例" → agent（定义表达风格）
       - `subject='user'` → 在定义 **user 是谁**：身份、背景、偏好（保持原貌，不要转译成 agent 指令）
         - 例：
           - "我是前端工程师" → user（定义用户身份）
           - "我偏好 Vue 生态" → user（定义用户偏好）
           - "我的母语是中文" → user（定义用户背景）
       - 两类可同时产出：用户说"我是前端" → 可写 **两条 candidate**（user: 用户是前端工程师 / agent: 回答时默认用前端语境举例）。让模型自行判断是否两条都写；description 里给清示例
       - 关键消歧义提示：第二人称"你"一律指 agent。用户说"你的名字叫 xxx" → **一条 candidate**（agent: 你叫 xxx，当用户直接用 xxx 称呼时，要明白是在叫自己）——这是在定义 agent 的身份，不是用户自称叫 xxx。
   - **反思 Job · `persona_extraction`**（`packages/agent/src/reflection/runner.ts`）
     - system prompt 重写：对每条 candidate 判断"**这条信息在定义 agent 还是定义 user**"，产出 `{candidates: [{content, subject, confidence, tags}]}`
     - 提示模型"同一批对话可以同时产 agent/user 两类 candidate，请分别列出；subject 判断只看'这条信息在定义谁'"
     - `parsePersonaOutput` 返回值新增 `subject` 字段
     - dedupe 的 key 从 `content` 改成 `(content, subject)` 组合键

3. **注入路径 · `PersonaSource`**（`packages/agent/src/context/persona.ts`）
   - 读取后按 subject 分两组，注入**两段** system message：
     ```
     # 关于你（agent）：以下是对你自己的定义，请据此设定身份、性格与行为方式
     以下条目定义了"你是谁"（来自过往对话沉淀，已由用户审核通过）。在本次对话中请据此扮演对应身份、保持对应性格与行为方式：
     - <agent-persona-1>
     - <agent-persona-2>

     # 关于用户：以下是对你交流对象的定义，请据此调整交流方式（术语层级、语气、举例风格等）
     以下条目定义了"用户是谁"（身份、背景、偏好，已由用户审核通过）。你可以在需要时自然引用（"据我所知你是..."），并据此选择合适的术语、举例与表达：
     - <user-persona-1>
     - <user-persona-2>
     ```
   - 两组各自 topK（建议 agent topK=10 / user topK=8）
   - 任一组为空则对应段不注入；两组都空则整个 Source 返回 null（与现有行为一致）

4. **审核 UI · `PersonaReviewBanner`**（`packages/ui/src/components/PersonaReviewBanner.tsx`）
   - 每条待审 candidate 前加 subject 标签徽章：`[关于你]` / `[关于用户]`（与注入段落标题保持一致）
   - 折叠态文案微调：`> N 条 Persona 定义待审核（关于你 / 关于用户）`
   - 不改审核动作语义，只是标签区分（视觉差异即可，不动后端）

**验收标准**
- [ ] `PersonaRecord.subject` 编译期必填；漏传会 TS 报错
- [ ] `remember_persona` tool schema 含 `subject` 且 required
- [ ] `PersonaSource` 注入时能看到两段 `# 关于你（agent）` / `# 关于用户` 且只在对应组非空时出现
- [ ] 反思 Job 至少在一次含"我是 X / 叫我 Y"的 visit 上能产出 subject 混合的 candidates 列表
- [ ] `PersonaReviewBanner` 每条 candidate 有 subject 徽章（`[关于你]` / `[关于用户]`）
- [ ] **不存在**任何 `subject ?? 'agent'` 兜底读取；老记录读出 subject undefined 时不做 fallback（TS 必填 + 产品未发布，放弃向下兼容）

**工作量**：~2h（集中在 schema / persona.ts / remember-persona.ts / reflection/runner.ts / banner.tsx 五个文件）

### 需求 2 · 时间维跨 visit 记忆检索 · `recall_memory` 扩参 + 新增 `list_recent_visits` tool

**是什么**
把"语义召回"与"时间维列清单"两种意图**拆成两个 tool**：
- `recall_memory` 保留为纯语义召回，扩参 `timeRange` / `domain` 做窗内过滤
- **新增** `list_recent_visits` tool，按 `timeRange` + `domain` + `limit` 直接列时间窗内的 visit_summary 清单（不走向量）

同时在 `RelevantMemorySource` 里做**自动路由**：粗判命中时间维元查询时，跳过向量召回直接按时间窗注入清单，让用户感知不到 tool 边界。

**为什么**
`recall_memory` 原本只基于语义向量，对"今天看了哪些文章"、"本周读了什么"这类**元查询**（内容无关、时间相关）天然失效。两个意图语义不同、参数形态不同、返回结构不同，合在同一个 tool 里会出现"`query` 是否必填"、"默认 limit 是多少"这类条件分支，既不利于 LLM 选型，也不利于单测。

**两个 tool 的好处**：每个 tool 的参数都是稳定必填集合，description 可以各自写清意图；LLM 在 tool 选择阶段按"按时间段列清单 → list_recent_visits；语义召回 → recall_memory"即可判断。自动路由分支则保证即便 LLM 没调 tool，系统也能在 `RelevantMemorySource` 侧就近补上时间窗清单。

**实施方案**

1. **`recall_memory` 扩参**（`packages/tools/src/definitions/recall-memory.ts`）
   - 新 schema（保留 `query` 必填、**不再有 `mode` 字段**）：
     ```json
     {
       "type": "object",
       "properties": {
         "query": {
           "type": "string",
           "description": "10-30 字自然语言，围绕核心实体/概念。用于'上次那个方案'、'我们之前聊的 agent loop'等**内容线索**查询",
           "minLength": 1
         },
         "timeRange": {
           "type": "string",
           "enum": ["today", "yesterday", "this-week", "last-week", "last-7-days", "custom"],
           "description": "可选时间窗过滤。向量召回结果进一步按该窗口做二次过滤"
         },
         "startTs": { "type": "integer", "description": "custom 起点（毫秒时间戳）" },
         "endTs":   { "type": "integer", "description": "custom 终点（毫秒时间戳）" },
         "domain":  { "type": "string", "description": "可选域名过滤，如 'github.com'" },
         "articleId": { "type": "string", "description": "可选 visit/article id 过滤（定位到某次具体浏览）" },
         "limit": {
           "type": "integer",
           "minimum": 1,
           "maximum": 20,
           "description": "返回条数，默认 3"
         }
       },
       "required": ["query"],
       "additionalProperties": false
     }
     ```
   - `execute` 只做一件事：`deps.recallSemantic({ query, timeRange, startTs, endTs, domain, articleId, limit })`。**不做隐式降级**、**不含 `time_query_unsupported` 分支**（该分支彻底删除——LLM 应该直接调 `list_recent_visits`，不再由 `recall_memory` 兜底提示）。
   - 老的 `mode: 'auto' | 'explicit'` 枚举**彻底删除**（如果仓库里还在）；不做双写兼容。
   - `RecallMemoryToolDeps` 简化为：
     ```ts
     export interface RecallMemoryToolDeps {
       recallSemantic(args: {
         query: string;
         timeRange?: TimeRangeKey;
         startTs?: number;
         endTs?: number;
         domain?: string;
         articleId?: string;
         limit?: number;
       }): Promise<{ hit: boolean; text: string; count: number }>;
     }
     ```
   - description 改写：
     ```
     从用户过去的浏览/对话记忆中按**主题/语义**召回内容，用于"上次那个 agent loop 方案"、
     "我们之前聊的兜底逻辑"等**内容线索**查询。可叠加 timeRange/domain/articleId 做窗内过滤。

     注意：如果用户问的是"今天/本周看了什么"、"昨天聊了啥"这种**时间维元查询**（内容无关），
     请改用 list_recent_visits，不要在这里硬塞 query。
     ```

2. **新增 `list_recent_visits` tool**（`packages/tools/src/definitions/list-recent-visits.ts`，新文件）
   - schema：
     ```json
     {
       "type": "object",
       "properties": {
         "timeRange": {
           "type": "string",
           "enum": ["today", "yesterday", "this-week", "last-week", "last-7-days", "custom"],
           "description": "时间窗口。custom 时必须同时提供 startTs/endTs"
         },
         "startTs": { "type": "integer", "description": "custom 起点（毫秒时间戳）" },
         "endTs":   { "type": "integer", "description": "custom 终点（毫秒时间戳）" },
         "domain":  { "type": "string", "description": "可选域名过滤，如 'github.com'" },
         "limit": {
           "type": "integer",
           "minimum": 1,
           "maximum": 50,
           "description": "返回条数，默认 20"
         }
       },
       "required": ["timeRange"],
       "additionalProperties": false
     }
     ```
   - `execute` 直接走底层 `memory.recall({ types: ['visit_summary'], timeRange: ..., domain, limit })`，返回结构化 `visits[]` 交给 LLM 自行叙述：
     ```ts
     export interface ListRecentVisitsToolDeps {
       listRecentVisits(args: {
         timeRange: TimeRangeKey;
         startTs?: number;
         endTs?: number;
         domain?: string;
         limit?: number;
       }): Promise<{
         count: number;
         visits: Array<{
           visitId: string;
           url: string;
           title?: string;
           domain?: string;
           summary: string;
           tags: string[];
           timestamp: number;
         }>;
       }>;
     }
     ```
   - description：
     ```
     按时间窗列出用户最近浏览过的页面摘要清单（visit_summary），用于"今天看了哪些文章"、
     "本周读了什么"、"昨天我们聊了啥"等**时间维元查询**。不走向量召回，直接按 timeRange
     取 visit_summary 列表并按时间倒序返回。

     如果是"上次那个方案"、"我们之前聊的 X"这种**内容线索**查询，请改用 recall_memory。
     ```

3. **time-query util 上移**（`packages/agent/src/context/time-query.ts`，新文件）
   - 把老的 `detectTimeScopedMetaQuery` 从 `packages/tools/src/definitions/recall-memory.ts` **移到**此处（避免 agent 反向依赖 tools）。tools 源文件中的原函数同步删掉，不留别名/re-export。
   - 同文件放 `resolveTimeRange(timeRange, now): { startTs: number; endTs: number }` 与 `TimeRangeKey` 类型（`today/yesterday/this-week/last-week/last-7-days/custom`）。
   - 这份 util **供三处共享**：
     1. `list_recent_visits` tool 的 runner
     2. `recall_memory` 的 `timeRange` 二次过滤
     3. `RelevantMemorySource` 的自动路由分支

4. **自动路由 · `RelevantMemorySource`**（`packages/agent/src/context/relevant-memory.ts`）
   - 在 `gather()` 入口、走原语义召回之前，先调用 `detectTimeScopedMetaQuery(ctx.userInput)`：
     - 命中 → 直接走 `memory.recall({ types: ['visit_summary'], timeRange: ..., limit: 10 })` 拼装成 system 段：
       ```
       # 相关历史记忆（按时间窗自动召回）
       用户问题含时间维关键词，以下是该时间窗内的 visit 摘要清单：
       1. [2026-04-28 · github.com] <title> — <summary> (tags: a, b)
       2. ...
       ```
     - 未命中 → 走原有链路（粗判 → aux 精判 → 向量）
   - 这条"系统侧隐式路由"与"LLM 侧显式调用 `list_recent_visits`"并列：系统帮用户省掉一次 tool 调用，LLM 在用户意图明确时也可显式调用；两者走同一条底层数据路径（`memory.recall({ types: ['visit_summary'], ... })`），复用同一份 `resolveTimeRange`。

5. **System Prompt 升级**（`packages/agent/src/` 里主 system prompt 源，当前在 `SystemPromptSource` 及各 builder）
   - 加一条行为守则：
     ```
     记忆检索的两个 tool 分工（不要混用）：
     - 按时间段列清单（"今天/本周/最近看了什么"、"昨天聊了啥"）→ list_recent_visits({ timeRange:'today' })
     - 语义召回（"上次那个方案"、"我们之前聊的 X"）→ recall_memory({ query:'...' })，可带 timeRange 做窗内过滤
     两条路径都空才回复"找不到相关记忆"。
     ```

6. **Tool 注册**（`packages/tools/src/definitions/index.ts` 及 `buildPhase2Tools`）
   - tool 总数 **+1**：`list_recent_visits` 作为新成员注册到 Phase2 工具集。
   - agent 装配层分别注入 `recallSemantic` 与 `listRecentVisits` 两个 runner（底层都命中 `memory.recall`，但对 tool 层分别暴露）。

**验收标准**
- [ ] `recall_memory` 参数 schema：`query` 必填，`timeRange/startTs/endTs/domain/articleId/limit` 均可选；**不再存在** `mode` 字段
- [ ] `list_recent_visits` tool 已注册；schema：`timeRange` 必填，`startTs/endTs/domain/limit` 可选
- [ ] `list_recent_visits({ timeRange:'today' })` 能直接拿到今日 `visit_summary` 清单（单测里不调用向量 runner）

### 需求 4 · 话题漂移 · 关键词触发小改进

**是什么**
在 `shouldIdentify(userMsgCount, interval)` 外加一条"**关键词漂移检测**"旁路：若当前 user 输入包含"换个话题 / 换话题 / 说点别的 / 不说这个了 / let's switch / new topic"等显式漂移词，**即使**未到 interval 周期，也立即触发主题重识别。

**为什么**
当前话题识别是固定每 4 轮周期，实测里用户主动说"换个话题聊 X"时，下一轮 SessionTopic 仍是旧话题，aux 要等到第 4 轮末才重识别 —— 用户感知到的"话题滞后"。

完整版（相似度阈值 / aux 判定漂移）工作量大、ROI 不确定，**v0.4.0 只做关键词触发**（0.5h），真机跑一段看效果，完整版留给 v0.5。

**实施方案**

1. **扩展 `shouldIdentify`**（`packages/agent/src/aux/session-topic.ts`）
   - 签名扩展：
     ```ts
     export function shouldIdentify(
       userMsgCount: number,
       opts?: { interval?: number; latestUserInput?: string },
     ): boolean;
     ```
   - "首条触发 + 每 interval 触发"的核心语义维持不变；在其上叠加一条新旁路：若 `opts.latestUserInput` 命中 `TOPIC_DRIFT_PATTERNS`，直接返回 `true`。
   - `TOPIC_DRIFT_PATTERNS`（放同文件或 `packages/agent/src/aux/topic-drift.ts`，视 developer 喜好）：
     ```ts
     const TOPIC_DRIFT_PATTERNS: ReadonlyArray<RegExp> = [
       /换个话题|换话题|说点别的|不说这个|聊点别的|来聊聊|我们聊聊|另一个话题/,
       /\b(let'?s (switch|change|talk about)|new topic|change the subject|different topic)\b/i,
     ];
     ```
   - 返回值语义不变（true/false），调用方无需改动分支逻辑。

2. **调用点调整**（找 sidebar / useStreamingChat 里调 `shouldIdentify(...)` 的地方）

   - 其它参数沿用

   - 负 case：`shouldIdentify(2, { latestUserInput: '这段代码什么意思' })` 返回 false（未到周期、无漂移词）
   - 原有 case 不破（旧签名 `shouldIdentify(count)` 等价 `shouldIdentify(count, {})`）

**验收标准**
- [ ] `shouldIdentify` 支持 `latestUserInput` 可选入参；命中漂移词立即返回 true
- [ ] 新增至少 2 条单测（命中 / 未命中）

## 修订记录

- **2026-05-07 · 实现后修订 · 规格补登**：`deepseek-v4-flash` 与 `deepseek-v4-pro` 两款模型
  的官方规格同步登记到能力表：
  - 上下文窗口：**1,000,000 tokens**（1M；此前能力表保守填 65536）
  - 单次最大输出：**384,000 tokens**（384K）
  落地点：`DEEPSEEK_MODEL_CAPABILITIES`（`packages/provider/src/deepseek/config.ts`）。
  同时在 `ModelInfo` 与 `GenericModelListItem.capability` 上新增可选
  `maxOutputTokens?: number` 字段以承载"最大输出能力上限"；Qwen 侧留空（未来按需回填）。
  该值仅作为**能力声明**，运行时默认 `max_tokens` 不自动撑到 384K，仍由 Provider /
  Agent 层保守决定，避免单次请求把配额打爆。UI 下拉 tooltip（`BasicTab` /
  `ProviderConfigForm`）检测到该字段后追加 "· max_out ≈ N tokens" 展示。
- **2026-05-07 · 实现后修订**：DeepSeek 官方模型清单同步更新——旧模型
  `deepseek-chat` / `deepseek-reasoner` / `deepseek-coder` 等**已全部下线**，当前线上仅
  两款 chat 模型：
  - `deepseek-v4-flash`（低成本快响应档）
  - `deepseek-v4-pro`（主力档）
  正文中出现的具体模型名（主要在 §3 / §4 / §7 / §10 示例与 AC）语义上应理解为
  "当前线上等价模型"的历史名称，代码与测试已以新模型名落地；请以实际代码为准。
  同时：`/models` 接口返回体**无 category / capability 字段**，`id` 即模型名，全部
  归为 chat 分类；`reasoning-delta` 链路保留但不再绑定特定模型（"思考/非思考"路径
  不再在 Payload 上显式区分，由模型自行决定是否返回 `reasoning_content`）。

### 1.3 目标（一句话）

> **让用户在配置页下拉里选到 "DeepSeek"、填 API Key 与模型（如 `deepseek-chat` / `deepseek-reasoner`）即可接入；同时明确支持"DeepSeek 主模型 + Qwen 向量模型"这种跨 Provider 组合；Provider 层抽出一层薄基类 `OpenAICompatibleProvider`，DeepSeek 作为它的一个实例仅需~100 行代码，未来再接第三家（OpenAI / 月之暗面 / Moonshot / 自托管）的边际成本同样被压到最低。**

### 1.4 用户价值

1. **更低的推理成本**：DeepSeek V3/R1 的价格带在中文开发者场景里性价比最优；降低用户跑主对话、反思 Job、Intent 精判的账单。
2. **R1 深度推理**：Doc Assistant 场景里"对比两篇论文的方法差异 / 解释复杂源码调用链"这类问题，R1 带 `reasoning_content` 的回答质量明显更好。`ChatChunk.reasoning-delta` 已就位，用户可以直接看到折叠的思考过程。
3. **向量模型零迁移**：DeepSeek 官方无 embedding 服务；通过"组合渠道"机制，老用户之前攒下的 `episodes_visit_summary`（基于 Qwen v3 embedding）不用重建索引。
4. **面向未来**：抽一次薄基类后，再接 OpenAI / Moonshot / Anthropic（严格说 Anthropic 协议不兼容 OpenAI，但 chat 基本结构可复用大半）只是"填表"，不必每家重写 250 行。

## 2. 用户故事 / 使用场景

### US-1 · 第一次配置：混搭（主力场景）

> 小瑾是老用户，主对话已经用 Qwen 很久，`episodes_visit_summary` 已经攒了几百条向量。她在 2026 年 5 月升级到 0.6.0-beta.2 后打开配置页，看到主 Provider 下拉里多了一项 "DeepSeek"。她**只切换主对话**到 DeepSeek：Provider 选 DeepSeek、`baseURL` 自动填 `https://api.deepseek.com`、`model` 填 `deepseek-chat`，API Key 粘进去。向量那一路她**保持不变**（Qwen `text-embedding-v3`，复用主 Provider 关掉——因为主 Provider 已经不是 Qwen 了，她需要在向量路单独填 Qwen 的 apiKey + baseURL）。保存后，新对话在 sidebar 里流式正常，老的记忆依然能被召回。

### US-2 · 深度推理

> 老王读 arXiv 论文，想试 `deepseek-reasoner`。主 Provider 切 DeepSeek、model 填 `deepseek-reasoner`，"启用思考模式"开关默认亮起（见 §4 的 UI 约定）。发起对话后，UI 上出现"思考过程"折叠块（已经复用 v0.5.x 落地的 `ThinkingBlock`），展开能看到完整 chain-of-thought。辅助 LLM 他仍用 Qwen Turbo 省钱。

### US-3 · 全 DeepSeek + Qwen 向量（最常见组合）

> 配置：`main = DeepSeek(deepseek-chat)` / `aux = DeepSeek(deepseek-chat)` / `embedding = Qwen(text-embedding-v3)`。配置页在向量一栏会给出一条提示："检测到主 Provider 为 DeepSeek。DeepSeek 暂无官方 embedding 服务；推荐向量路选择 Qwen `text-embedding-v3`，已为你填好默认 baseURL，请粘贴 Qwen API Key。" 用户点击"使用推荐"按钮，一键填入即可。

### US-4 · 零停机切换，老记忆不丢

> 用户从"全 Qwen"切到"主 DeepSeek + 辅 DeepSeek + 向量 Qwen"后，老的 `episodes_visit_summary`（Qwen v3，1024 维）**不被触碰**。`recall_memory` 继续命中，`list_recent_visits` 继续正常。

### US-5 · 未来再接一家 Provider（非本期交付，但是抽象验证）

> Developer B 想接 Moonshot / Kimi。他打开 `packages/provider/src/`，照着 `deepseek/index.ts`（本期产物，预计 80-120 行）写一个 `moonshot/index.ts`，在 provider 注册表里加一行，配置页下拉自动出现 Moonshot 选项，无需改动 UI 或装配层——**本期抽象的验证标准**。

## 3. 功能范围

### 3.1 In-Scope（本期必做）

#### 3.1.1 Provider 适配器（代码层）

1. **新增 DeepSeek Chat Provider**（LLM 能力）：
   - 流式对话（text-delta）
   - Tool calling（function call / tools 格式，DeepSeek 兼容 OpenAI schema）
   - 思考模式：`deepseek-reasoner` 的 `reasoning_content` 通过 `ChatChunk.reasoning-delta` 流出（复用现有 `reasoning-delta` 分支）
   - Usage 统计（`promptTokens` / `completionTokens`）归一化到现有 `ChatChunk.usage`
   - 错误归一化（401 / 429 / 404 / 5xx / 网络 / Abort）走现有 `ProviderError`
   - 支持 `signal?: AbortSignal` 取消
2. **不做 DeepSeek 自研 embedding**：DeepSeek 官方无 embedding 服务；若用户在 embedding 下拉选择 DeepSeek，**不提供该选项**（从下拉中屏蔽，而非"选了再提示"，理由见 §4.3）。
3. **DeepSeek 模型能力表**（类似 `QWEN_MODEL_CAPABILITIES`）：至少覆盖 `deepseek-chat` / `deepseek-reasoner` 两款，标注 `contextWindow` / `supportsReasoning` / `supportsTools`。
4. **DeepSeek list-models**：调用 `GET {baseURL}/models` 拉列表（DeepSeek 也遵循 OpenAI 兼容，实测可用）。本期可用共享的 `listOpenAICompatibleModels`（见 §5）而非单独为 DeepSeek 写一份。

#### 3.1.2 Provider 层抽离（本期必做的基类）

见 §5 详细方案；一句话概括：抽出 `OpenAICompatibleProvider` 基类 + 共享工具，Qwen 与 DeepSeek 都基于其实现，Qwen 保留"思考模式透传 extra_body"这一特化。

#### 3.1.3 组合渠道（主 + 向量跨 Provider）

- Schema 上**已经支持**（`MAIN_PROVIDER_CONFIG` 与 `EMBEDDING_PROVIDER_CONFIG` 分开存）——本期做**产品化**（UI 提示 / 推荐 / 校验 / 默认值）。详见 §4。

#### 3.1.4 配置 schema 扩展

- `ProviderKind = 'qwen' | 'deepseek'`（扩展联合）
- `EmbeddingProviderKind = 'qwen-embedding'`（**不扩**；DeepSeek 不接入 embedding 这条路）
- `DEFAULT_DEEPSEEK_PROVIDER_CONFIG` 常量（baseURL 默认 `https://api.deepseek.com`，model `deepseek-chat`，enableThinking 默认 false；`deepseek-reasoner` 由用户显式切）
- 保存时的 zod schema（`OptionsForm.tsx`）改为按 `kind` 分支校验

#### 3.1.5 UI / 配置页

- Provider 下拉：主 / 辅两组新增 "DeepSeek" 选项（embedding 组**不**加）
- 切换 Provider 时：`baseURL` 自动换为新 Provider 的默认值（若当前就是旧默认值，否则保留用户自定义）；`apiKey` 不自动清空；`model` 切到新 Provider 的推荐值（`deepseek-chat`）
- `ProviderConfigForm` / `BasicTab` 的"拉取模型列表"按钮按当前 `kind` 路由到对应的 list-models 实现
- embedding tab 增加"组合 Provider 推荐"提示条：当**主 Provider 为 DeepSeek 且 embedding 仍勾选"复用主 Provider"时**，显眼提示 + "使用 Qwen `text-embedding-v3` 作为向量模型"的一键按钮
- 测试连接按钮适配 DeepSeek（复用 `POST /chat/completions` 逻辑即可）

#### 3.1.6 装配层

- `sidebar/bootstrap.ts` / `offscreen/index.ts` 改为按 `kind` 反射到对应 Provider 类；错误路径与 Qwen 对齐
- 反思 Job（offscreen）在 `aux` 为 DeepSeek 时也能正常跑（当前反思 Job 是非流式/结构化输出，走常规 chat 即可）

#### 3.1.7 文档

- `docs/CHANGELOG.md` 新增 `## [0.6.0-beta.2]` 段落
- `docs/ROADMAP.md` 版本路线表新增 `0.6.0-beta.2 · DeepSeek Provider + Provider 抽离` 一行
- `README.md` 的"支持的模型 / Providers"章节同步（如无该章节，本期新增）
- 本需求文档定稿后，历史 `v0.5.1-deepseek-provider.md` 顶部加一行"→ 已并入 0.6.0-beta.2"指针

### 3.2 Out-of-Scope（本期明确不做）

1. **OpenAI / Anthropic / Moonshot / Ollama 的任何一家**。基类抽完不等于顺手加三家。
2. **DeepSeek 自研 embedding**。官方未发布；若未来官方发布，再评估。
3. **Prefix / FIM / Beta 接口**（DeepSeek 特有功能）。
4. **Provider 级 token 用量看板**（ROADMAP v0.2.5+ 的独立项）。本期 usage 归一化保持与 Qwen 一致的字段结构，为看板落地留路径。
5. **Anthropic 协议兼容适配**（其协议与 OpenAI 差别比 DeepSeek 大，不在基类规划内）。
6. **配置迁移脚本**：新增 `kind='deepseek'` 是**纯加法**，老用户 Qwen 配置原样工作，**不写迁移脚本**。
7. **embedding 下拉的 DeepSeek 选项**：本期**直接不加**（与历史 PRD v0.5.1 的口径不同；见 §4.3 的理由）。
8. **`ChatChunk` 契约扩展**：已在 v0.5.x 完成（`reasoning-delta` 已存在），本期**不动**跨层契约。
9. **Provider 插件化注册机制**（允许运行时动态注入 Provider）。本期只做代码级的 Provider Registry 常量，不支持热插拔。

## 4. 组合渠道方案（主 + 向量跨 Provider）

这是本期与 v0.5.1 历史 PRD **最显著的产品化差异**。

### 4.1 现有机制回顾

代码层面，`STORAGE_KEYS.MAIN_PROVIDER_CONFIG` 与 `STORAGE_KEYS.EMBEDDING_PROVIDER_CONFIG` 本就是独立存的；`ProviderConfigOrRef<T>` 的 `{ useMain: true }` 只是一种"复用"语法糖。所以**跨 Provider 组合在数据层已经支持**——问题在产品形态（UI 没引导 / 没推荐 / 没校验）。

### 4.2 产品形态（本期要落地）

#### 4.2.1 UI 展示逻辑（`MemoryTab.tsx` embedding 卡片）

当**主 Provider.kind === 'deepseek'** 且 embedding 仍是 `{ useMain: true }` 时：

- 卡片顶部显眼显示 **警告 + 推荐**：
  > ⚠️ 当前主 Provider 为 **DeepSeek**，DeepSeek 暂无官方 embedding 服务。
  > 记忆层向量召回功能依赖 embedding，建议切换到 **Qwen `text-embedding-v3`**（与主对话解耦，不影响聊天质量；老用户的历史记忆索引也由 Qwen 建成，可无缝继续使用）。
  > [**一键使用推荐配置**] [了解更多](链接到本文档或 README)
- "一键使用推荐配置"按钮：
  - 关闭 `useMain` 开关
  - 填入 `{ kind: 'qwen-embedding', baseURL: 'https://dashscope.aliyuncs.com/compatible-mode/v1', model: 'text-embedding-v3', dimension: 1024, apiKey: '' }`
  - 把 apiKey 输入框聚焦，提示用户粘贴

#### 4.2.2 UI 展示逻辑（辅助 Provider aux 卡片）

- aux 仍允许 `useMain`，无强制警告。用户如果主 = DeepSeek、aux = useMain，那 aux 自然也是 DeepSeek。

#### 4.2.3 默认值

- 新安装用户：`DEFAULT_MAIN_PROVIDER_CONFIG.kind = 'qwen'` 保持不变（主 Provider 默认仍推荐 Qwen，因为它 embedding / chat 都全，新手上手最简单）。
- 老用户升级：配置原样保留，不触发任何自动切换。

#### 4.2.4 保存时校验

- 保存前的 zod 校验改为按 `kind` 分支：
  - `kind: 'qwen'` → 校验 `apiKey / baseURL / model / 可选 enableThinking`
  - `kind: 'deepseek'` → 校验 `apiKey / baseURL / model / 可选 enableThinking`（deepseek-reasoner 时隐式等同）
- **软校验**（非阻塞）：当主=DeepSeek 且 embedding useMain=true 时，保存前弹 `Modal.confirm`："向量召回需要 embedding 服务，而 DeepSeek 无此能力。确认保存？记忆召回可能不工作。" — 用户点"确认"仍可保存（为自托管兼容场景留余地），点"改成推荐配置"则回到 §4.2.1 的一键填入。

#### 4.2.5 降级链路

沿用现有 offscreen 的降级（见 `offscreen/index.ts:155-177`）：embedding 初始化失败时 `embeddingProvider = null`，`RelevantMemorySource` 退化为关键词召回，**不影响主对话**。本期不改。

### 4.3 为什么"embedding 下拉不放 DeepSeek"（与历史 PRD v0.5.1 口径差异）

v0.5.1 的 AC-UI-4 让 DeepSeek 出现在 embedding 下拉里、选中后靠提示引导用户别用。本期改为**直接不放**，原因：

1. DeepSeek 官方端点调用 `/embeddings` 会返回 404，保留选项只会制造更多支持工单。
2. 自托管兼容场景（用户挂一个 OpenAI 兼容网关指向某个开源 embedding 模型）走"Qwen 兼容端点"即可——`qwen-embedding` 实际上就是"OpenAI 兼容 `/embeddings` 客户端"，baseURL 自由填。若未来需要把"自托管 OpenAI 兼容 embedding"作为独立 `kind` 暴露，单独立项，不在本期。
3. 简化 UI 逻辑：embedding 下拉只有 Qwen 一项即可（后续接入 OpenAI 等真正有 embedding 的 Provider 时再扩）。

## 5. 重构 / 抽离建议（本期必做）

### 5.1 目标与边界

**目标**：让新增一家 OpenAI 兼容 Provider 的成本 ≤ 120 行业务代码 + 1 条注册表登记 + 1 份 UI 默认值。

**边界**：不做过度设计。基类只抽"当前 Qwen 实装里和 DeepSeek 100% 重合"的部分；特化（Qwen 的 `enable_thinking` extra_body、DeepSeek 的 `reasoning_content` 自动打开）保留在各自子类里。

### 5.2 具体目录与文件拆分

#### 5.2.1 新增共享基类与工具（`packages/provider/src/openai-compatible/`）

```
packages/provider/src/openai-compatible/
├── provider.ts          · OpenAICompatibleProvider 基类（chat 流式 + tool calling + ChatMessage→CoreMessage + tool schema 转换）
├── embedding.ts         · OpenAICompatibleEmbeddingProvider 基类（/embeddings 分批 + 错误/维度校验）
├── list-models.ts       · listOpenAICompatibleModels + classifyByIdPrefix（可注入自定义分类规则）
├── normalizer.ts        · AI SDK stream part → ChatChunk 归一化（现 qwen/normalizer.ts 原样迁移）
├── config.ts            · 共享的 zod schema 片段（baseURL + apiKey + model）+ joinUrl / safeReadText / safeParseJSON 小工具
└── index.ts             · barrel export
```

#### 5.2.2 Qwen 子类瘦身（`packages/provider/src/qwen/`）

- `qwen/index.ts`：继承 `OpenAICompatibleProvider`，只保留 `enable_thinking` 的 `providerOptions.openai.extra_body` 透传特化，以及 `getModelInfo()` 里"`supportsReasoning && enableThinking`"的与运算。
- `qwen/config.ts`：保留 `QWEN_MODEL_CAPABILITIES` 模型能力表 + `qwenProviderConfigSchema`（基于共享 schema 扩展 `enableThinking`）。
- `qwen/embedding.ts`：继承 `OpenAICompatibleEmbeddingProvider`，仅保留 `text-embedding-v2/v3` 的 `CAPABILITY_TABLE` 与默认 `maxBatchSize=25`（千问硬限制）。
- `qwen/list-models.ts`：调用共享 `listOpenAICompatibleModels`，只传入 Qwen 专属的 `classifyQwenModel`（保留现有分类细则）。
- `qwen/normalizer.ts`：直接删（或变成 `export * from '../openai-compatible/normalizer'`），内容迁入基类。

#### 5.2.3 DeepSeek 新增（`packages/provider/src/deepseek/`）

```
packages/provider/src/deepseek/
├── index.ts        · DeepSeekProvider extends OpenAICompatibleProvider（~60 行：覆盖 getModelInfo + 处理 enableThinking 在 deepseek-reasoner 下的语义）
├── config.ts       · deepSeekProviderConfigSchema + DEEPSEEK_MODEL_CAPABILITIES（至少含 deepseek-chat / deepseek-reasoner）
└── list-models.ts  · 薄包装（复用 listOpenAICompatibleModels + DeepSeek 专属分类——实际上只有 chat 一种 kind，简化版即可）
```

**预估新增代码**：~150 行（含测试骨架），其中业务代码 ~80 行。

#### 5.2.4 Provider Registry（`packages/provider/src/registry.ts`）

用一个常量表打通 `kind → {构造函数 / 默认配置 / list-models 函数 / 显示名}` 的映射：

```ts
// 伪代码示意，不是实现
export const PROVIDER_REGISTRY = {
  qwen: {
    displayName: '千问 Qwen（阿里云百炼）',
    LLMClass: QwenProvider,
    defaultConfig: DEFAULT_QWEN_CONFIG,
    listModels: listQwenModels,
    // embedding 能力标志：有 / 无 / 兼容端点自托管
    embedding: { kind: 'qwen-embedding', EmbeddingClass: QwenEmbeddingProvider },
  },
  deepseek: {
    displayName: 'DeepSeek',
    LLMClass: DeepSeekProvider,
    defaultConfig: DEFAULT_DEEPSEEK_CONFIG,
    listModels: listDeepSeekModels,
    embedding: null, // 明确标注无 embedding
  },
} satisfies Record<ProviderKind, ProviderRegistryEntry>;
```

- 装配层（`sidebar/bootstrap.ts` / `offscreen/index.ts`）通过 `PROVIDER_REGISTRY[config.kind].LLMClass` 实例化。
- UI（`BasicTab` / `ProviderConfigForm`）的下拉 options / 拉模型函数 / 默认 baseURL 全部读 registry。
- 增加新 Provider 时**唯一**需要改的地方：新增目录 + 在 registry 加一行 + 如需则在 `ProviderKind` 联合里加字面量。

#### 5.2.5 UI 改造要点

- `BasicTab.tsx`：Provider 下拉 options 改从 `Object.entries(PROVIDER_REGISTRY)` 渲染；拉模型按钮的 handler 改为 `PROVIDER_REGISTRY[main.kind].listModels(...)`；切换 Provider 时 baseURL 按 `defaultConfig.baseURL` 替换。
- `ProviderConfigForm.tsx`：同上；embedding mode 下，Provider 下拉过滤 `embedding !== null` 的条目（因此只会看到 Qwen）。
- `OptionsForm.tsx`：zod 校验改为 `z.discriminatedUnion('kind', [...])`，每个 kind 一个 schema，从 registry 汇总。

#### 5.2.6 错误归一化

- `packages/provider/src/openai-compatible/errors.ts`：封装 `mapHttpErrorToProviderError(status, bodyText)` — 把 401 / 429 / 404 / 5xx 映射到统一 `ProviderError` code（`AUTH_ERROR` / `RATE_LIMITED` / `MODEL_NOT_FOUND` / `UPSTREAM_ERROR` / `NETWORK_ERROR`）。`qwen/embedding.ts` 现有的 `EMBEDDING_HTTP_ERROR` 等保留向后兼容 code，但内部 throw 处改为共享函数。

### 5.3 为什么这个抽象"不过早"

- 已有 Qwen 实现 250 行 + embedding 200 行 + list-models 250 行，**具体形状已跑通**。
- 本期再接第二家同协议 Provider（DeepSeek），**正好是抽象的第一次复用**，形状基本稳定。
- 抽完后再接第三家（Moonshot / OpenAI），可以继续打磨；若届时发现基类不合适，Qwen / DeepSeek 两个子类仍可各自开小灶（基类已经把"通用 60%"固化）。
- 与 v0.5.1 PRD R-3 "不抽象"的分歧在于：v0.5.1 把 DeepSeek 当作一次性落地估算工时 6-10h；本期把它定位为"抽象 + DeepSeek"，工时追加到 14-20h，但**接下一家 Provider 的工时骤降**。用户明确要求"新增代码量最小化"——这就是答案。

## 8. 风险与未决问题

### R-1 · DeepSeek Tool Calling 的流式分块差异（中）

- DeepSeek 虽然兼容 OpenAI 协议，但 `tool_calls` 在流式分块中的切分位置、`function.arguments` 的 delta 拼接策略、`finish_reason` 枚举值是否带 `tool_calls` 等细节可能与 Qwen / OpenAI 存在小差异。
- **应对**：Developer 实装时先跑一次真机抓包；`normalizer.ts` 现有的 `tool-call` / `tool-call-streaming-start` / `tool-call-delta` 分支已有兜底，预计不用改契约。若发现必须改 `ChatChunk`，回到 PM 单独走变更。
- **需 PM 拍板**：无；发现问题再议。

### R-2 · `deepseek-reasoner` 的 `reasoning_content` 字段位置（低）

- AI SDK v4 的 `@ai-sdk/openai` provider 对 `reasoning_content` 的识别依赖 provider 声明；DeepSeek 走 `createOpenAI({ baseURL: 'https://api.deepseek.com' })` 时，AI SDK 是否会自动把 `reasoning_content` 识别为 `reasoning` part 需要真机验证。
- **应对（已有预案）**：
  - **方案 A**：AI SDK 自动识别 → `normalizer.ts` 的 `reasoning` / `reasoning-delta` 分支直接生效，零改动。
  - **方案 B**：AI SDK 不识别（把 `reasoning_content` 当成扩展字段丢了）→ 在 `DeepSeekProvider` 里自己拦截 raw SSE 或通过 AI SDK 的 `providerOptions.openai` 开启 `extra_body`；若 AI SDK 完全封装到无法拦截，改走 `fetch` 裸流 + 自写 SSE 解析（放弃 `@ai-sdk/openai`，仅 DeepSeek 特化）。
- **需 PM 拍板**：无；Developer 真机验证后选 A/B。若落到 B 路径，工时 +2h。

### R-3 · Qwen 老用户升级时 UI 的 kind 下拉"默认值漂移"（低）

- 现有 Qwen 老用户的 `MAIN_PROVIDER_CONFIG.kind='qwen'` 已落地；本期把 kind 下拉 options 改为 registry 驱动后，Qwen 依然是第一个选项（新安装用户默认值）。不会自动改老用户的值。
- **需 PM 拍板**：无。

### R-4 · Provider Registry 不足以表达"能力矩阵"（中）

- 本期 registry 只打了最薄的映射（构造函数 / 默认配置 / list-models / 有无 embedding）。未来若需要表达"是否支持 reasoning / 是否支持视觉 / 是否支持流式 tool call / 价格区段"等细粒度能力矩阵，registry 需扩字段。
- **应对**：本期 registry 允许扩字段（字段为可选，向后兼容）；需要时再加。
- **需 PM 拍板**：本期不扩，默认接受。

### R-5 · DeepSeek API 限速导致真机抽样的测试不稳（低）

- DeepSeek 免费层有速率限制；CI / Developer 本地跑真机 e2e 时容易遇 429。
- **应对**：e2e 测试默认走 mock（`msw` 或 `fetch-mock`），只在本地 `pnpm test:live` 配环境变量 `DEEPSEEK_API_KEY` 时跑真机。
- **需 PM 拍板**：无。

### R-6 · 老用户保存路径的"discriminatedUnion" 校验变严（低）

- `OptionsForm` 改为 `z.discriminatedUnion('kind', ...)` 后，如果老用户 localStorage 中意外存了 `kind` 为空的脏数据，加载时可能通不过。
- **应对**：`storage.get` 返回后走现有 `{ ...DEFAULT_MAIN_PROVIDER_CONFIG, ...mainStored }` 合并，缺 kind 时兜底为 `'qwen'`；bootstrap / OptionsForm 都走这段。
- **需 PM 拍板**：无。

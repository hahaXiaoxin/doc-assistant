# Provider baseURL 自定义时 host_permissions 未动态申请

- **ID**: bugs-006
- **状态**: open(v0.4.0 起以"全开 `<all_urls>`"绕过,但留下安全口子待收口)
- **优先级**: P2
- **来源**: `docs/ROADMAP.md` · §6 + TROUBLESHOOTING §4(v0.1.1 遗留)
- **创建日期**: 2026-05-24

## 现象
用户在配置页填入自定义 Provider baseURL(支持 OpenAI / Anthropic / Ollama / 自部署等)时,manifest `host_permissions` 不会随之动态申请。v0.4.0 起为兼容任意 baseURL 把 `host_permissions` 直接放成 `["<all_urls>"]`,问题被绕过但安全面被显著放大。

## 复现
- 用户在配置页填一个 manifest 未声明的域名作为 baseURL。
- 在浏览器扩展商店(CWS)审核标准下,`<all_urls>` 是高风险声明,会被审核员要求收紧。

## 根因
- MV3 下 content script 跨源仍受 CORS,必须通过 `host_permissions` 声明豁免(详见 TROUBLESHOOTING §4)。
- `chrome.permissions.request({origins: [...]})` API 可在运行时申请,但当前 bootstrap 没有用。
- 短期为方便用户,直接 `<all_urls>` 全开;长期需要切回"按需申请 + 用户授权"。

## 期望行为
- 用户在配置页保存新 baseURL 时,触发 `chrome.permissions.request` 动态申请该 host。
- 用户拒绝时给明确提示("不授权该域将无法发请求")。
- manifest 默认 `host_permissions` 收回到内置受信域名(Qwen / DeepSeek / OpenAI 等已知列表),自定义 baseURL 走动态申请通道。
- CWS 审核 README / 隐私政策同步。

## 备注
- 与 ideas-009(WebFetch)的"manifest 权限策略重审"是同一根问题 — 一并处理可避免重复劳动。
- TROUBLESHOOTING §4 已记录原始决策"豁免 dashscope 域";v0.4.0 全开是过渡解。
- 修改 manifest 后需要在 `chrome://extensions` 手动刷新扩展一次,已打开 tab 也需刷新。

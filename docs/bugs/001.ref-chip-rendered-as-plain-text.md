# 划词引用在气泡里降级为纯文本

- **ID**: bugs-001
- **状态**: open
- **优先级**: P1
- **来源**: `docs/backlog.md` · B-005(可见 bug + 可扩展性铺路)
- **创建日期**: 2026-05-24

## 现象
- 在页面划词点"引用到 Doc Assistant" → 输入框出现紫色 chip(正常)。
- 发送后,在聊天气泡里,该 chip **变成 `<ref id="..." url="...">原文</ref>` 这样的伪 HTML 字符串**,失去 chip 视觉。
- 历史消息回看同样降级为纯文本。

## 复现
1. 任何文章页打开 sidebar。
2. 划一段文字 → 点选区悬浮 toolbar 的"引用到 Doc Assistant"。
3. 在输入框确认 chip 形态后回车发送。
4. 观察气泡内引用部分(应是 chip,实际为纯文本)。

## 根因
- 输入框侧:Lexical 自定义 `ReferenceNode`(`packages/ui/src/editor/nodes/ReferenceNode.ts`)+ `ReferenceTag` 渲染浅紫 chip,正常。
- 提交链路:`serializer.ts` 把 chip 拍扁为 `<ref id="..." url="...">text</ref>` 字符串,作为 `userInput` 走 LLM。
- 气泡渲染:`MessageBubble` 走 `react-markdown` + `remark-gfm` + **`rehype-sanitize` 默认白名单**;`<ref>` 既不在 GFM 语义里也不在 sanitize 白名单 → **降级纯文本**。
- 历史消息:落盘的也是这串 `<ref>` 字符串,任何修复必须同时保证老消息可还原 chip。

## 期望行为
- 输入框与气泡 chip 视觉一致(颜色 / hover / 截断)。
- 老历史消息(已存的 `<ref>` 字符串)向后兼容,不丢、不降级、不变纯文本。
- markdown 与 chip 共存,不互相打断。
- LLM 侧 `userInput` 仍含 `<ref>` 或等价语义,`referenceTagSource` 不变。

## 备注
- 修复方向归口到 ideas-004(`ContentRef` 数据模型 + `RefChip` UI 组件升级)。
- 安全红线:`rehype-sanitize` 白名单不得无差别放开,新自定义元素显式注册。
- 同时是富媒体扩展(图片 / DOM 快照 / PDF 片段)的铺路 — 现在做对一次,日后每加一种媒体不必重构 UI。

# Persona top-k 截断 · 长尾审核通过的记忆永远进不了 prompt

- **ID**: bugs-004
- **状态**: open
- **优先级**: P1
- **来源**: `docs/backlog.md` · B-007-B
- **创建日期**: 2026-05-24

## 现象
用户原话:**"目前的 persona 记忆只会选取 top-k 来加入,这就导致了可能有记忆加入不了。应该要给对应的模型提供相应的工具,让其整合 persona 记忆,例如重复的可以合并。同样的,当每次要新增一条的时候,也应该看看当前是否已经有了相似的记忆。"**

用户审核通过的 persona 数量超过 `agentTopK=10 / userTopK=8` 之后,**置信度/时间排序之外的长尾条目永远进不了 prompt**,即使它们被审核通过、是正确的。**审核通过 ≠ 真的被用上**,深层信任问题。

## 复现
1. 长期使用累积 ≥ 11 条 agent persona / ≥ 9 条 user persona(数月真机使用即可达到)。
2. 观察 LLM 收到的 PersonaSource system 段:置信度低 / 时间久的条目缺席。
3. 同时存在同义不同词条目共占 slot(例:"你说话要简洁" + "回答保持简洁、不加客套")。

## 根因
- **截断**:`packages/agent/src/context/persona.ts` L38–L68 硬截 `agentTopK=10 / userTopK=8`,排序键 `confidence DESC, updatedAt DESC`。
- **dedupe 太严**:`runPersonaExtraction` 的 dedupe(L285–L287)只做**严格 `content` 字符串相等 + subject 相等**,语义相同措辞不同的条目并存,加剧 top-k 挤压。
- **无自整理机制**:既无"新增前查相似",也无"合并重复/相似"批处理 tool。

## 期望行为
- 不无脑提高 topK(打爆 prompt + 带噪音)。
- 写入侧新增前相似度检查(同 subject)、合并批处理工具 `merge_personas(ids[])`。
- 合并后 PersonaRecord 保留来源 ID 链路(可回溯误合并)。
- 通过"整合降数量"让长尾被语义归并到头部,自然进入 prompt。

## 备注

- 不无脑改 top-k;不动已有老数据(存量整理留给"高级用户手动触发"按钮)。
- 待用户拍板:相似度阈值是否暴露 UI / 是否提供"高置信度自动合并"开关。

# 召回粗判 RECALL_PATTERNS 升级

- **ID**: ideas-016
- **状态**: proposed
- **优先级**: P2
- **来源**: `docs/ROADMAP.md` · v0.2.5+ "上下文分层 · 剩余工作"(2026-04-24 用户反馈)
- **创建日期**: 2026-05-24

## 背景
`packages/agent/src/context/recall-triggers.ts` 当前只有约 6 个中英正则("上次/之前/还记得/that topic"),且是召回链路的**第一道 AND 门** — 漏判则整条链路(精判 → 向量 → 邻居拼接)全不执行。漏报典型:
- "这个方案跟那个比起来..."(省略"之前"无时间锚点但实际引用历史)
- "你觉得这个实现好还是我们说的那个好"("我们说的那个"未入模式)
- 纯语义指代("它"、"这个想法"、"之前提过的那种写法")完全无模式可覆盖

设计原意是"宁可误报不漏报",但词表实际偏保守。精判每次 ~200–800ms,**多让 20% query 过门换高召回率是划算的**。

## 提案 / 三条优化路径(成本递增)
- **路线 A · 扩充正则词表**(~30 分钟,首选):
  - 中文补:单字"那"语境门、"之前说的"变体、"我记得..."起手式
  - 英文补:`I think we`、`earlier you said`、`that thing`、`back then`
  - 指代词独立粗触发(`/\b(那|这|it|that|this).{0,5}(个|方案|实现|想法|thing|approach)\b/`,user input >20 字时生效)
- **路线 B · 启发式**(~1h):无时间词但用"它/那 + 抽象名词"也触发,把准确性压力交给精判。
- **路线 C · 开源短文本分类 / 直接合并精判**(~2–4h):
  - C3 候选**最简单最鲁棒**:取消粗判,每条 user input 都送 aux 精判 + 24h LRU 缓存避免短期重复;按 ¥4/百万 tokens × ~5 token/次 ≈ 50w 次仅花 ¥1。

**推荐路径**:**先做 A**,真机跑一版看数据;若仍不满意评估 C3(最彻底)。

## 验收标准
- [ ] 收集 50–100 条真机 query,人工标注 should_recall 真值。
- [ ] 路线 A 落地后漏报率 < 10%。
- [ ] 误报率不显著上升(精判会兜底)。

# Prompt Caching 策略 · 重排上下文以最大化前缀命中

- **ID**: ideas-018
- **状态**: proposed
- **优先级**: P1
- **来源**: 用户反馈 · 自查 `packages/agent/src/agent.ts#composeMessages` / `packages/agent/src/context/index.ts` / `packages/provider/src/openai-compatible/sse-chat.ts`
- **创建日期**: 2026-05-27

## 背景

当前架构的核心思路是「**每一次对话都在重组 AI 所需的上下文**」——通过 `ContextSource` 数组(`buildDefaultSources` 返回 7 条 source),在每轮请求时按 priority 重新计算并拼装,目标是减少注入到 prompt 的冗余 token。

但这个思路**与上游模型的 prompt caching 机制是冲突的**。DeepSeek、Qwen 等 OpenAI 协议端点都已默认开启 **automatic prefix caching**:

- **DeepSeek** · 自动按 KV-Cache 命中,以 64 token 为最小单元;命中部分**收费 0.5 折**(命中价 = miss 价 / 10),`usage` 里返回 `prompt_cache_hit_tokens` / `prompt_cache_miss_tokens`,**全免操作、纯赚**。
- **Qwen / Bailian** · 部分模型支持 Context Cache(需在请求里显式开启 `enable_cache` 或走"上下文缓存"商品),命中后 prompt 计费同样大幅降低。

**结论**:当前我们既没有为 cache 命中做结构优化,也**没有**在 `extractUsage` 里读 cache 字段——既看不到命中率,也没有为命中铺路。两件事是有冲突的:**减少 token 输入** vs **保持稳定前缀以命中 cache**——后者在 DeepSeek 体系里几乎是"白送"的省钱路径,值得专门评估并融入策略。

## 当前 prompt 结构诊断

`Agent.composeMessages` 把 source 按 priority 降序展开,得到的实际消息结构如下(数字为 `priority`):

```
[system]   system-prompt          (100)  · 静态:固定的 system prompt + 工具使用规范
[system]   reference-tag           (70)  · 动态:用户当前轮的 ref chip → 每轮变
[system]   persona                 (60)  · 动态:listPersonas(confirmed) → top-K(agent=10/user=8) by confidence,desc
[system]   session-topic           (55)  · 半动态:每 3-5 轮辅助 LLM 识别一次,会变
[system]   working-memory          (50)  · 高频动态:TODO / ActiveGoal,几乎每轮变
[system]   relevant-memory         (40)  · 高频动态:按 query 召回,逐轮重算
[...messages] chat-history         (10)  · 末尾追加的真实历史 + tools 列表
[user]     当前用户输入
```

外加请求里的 `tools: ToolDefinition[]`,以及 `loop.ts` 最后一轮兜底**不带 tools**。

### 这个结构对 cache 命中有多差?

prefix caching 本质是**字节级前缀匹配**——只要前缀有一个字节抖动,后面的全部 cache 都失效。把上面 7 条对照看:

| 段位 | 稳定性 | 影响 |
| --- | --- | --- |
| system-prompt | **完全静态** | ✅ 唯一稳定的"打底" |
| reference-tag | 每轮变(用户引用不同) | ❌ 在 system-prompt 后**第一个**动态段——直接把后面所有 cache 杀掉 |
| persona | top-K + confidence 排序,新增/审核会变 | ❌ 即使内容没动,排序变了也 miss |
| session-topic | 阶段性变 | ❌ 触发刷新就 miss |
| working-memory | 几乎每轮变 | ❌ |
| relevant-memory | 每轮按 query 重算 | ❌ |
| chat-history | 累积型,**前缀稳定**(只往末尾追加) | ✅ 本来是天然 cache 友好——可惜被前面动态段堵了 |

**真实命中率(估计)**:cache 实际只能覆盖到 system-prompt 末尾,在引用 chip 出现的第一轮就只能命中 system-prompt 自己——DeepSeek 64 token 最小单元下,如果 system prompt < 64 token 甚至完全无法 cache。**长 chat history 的天然 cache 优势被前面的动态 system 段完全浪费了。**

此外:

- `loop.ts` 最后一轮兜底临时 push `LAST_TURN_SYSTEM_HINT` 并去掉 `tools`——一会话 1 次的 cache miss,可接受。
- `chatParams.tools` 字段如果序列化顺序不稳定(例如 Map 迭代),也会让 tools schema 的字节流变化。当前从数组直接传,顺序应稳定,但需要测试验证。
- `assistantMsg.reasoning` 已按 DeepSeek 协议在多轮里回传,这点是 cache 友好的,**保留**。

## 提案

### 目标

让"稳定的内容铺前缀,动态的内容贴当前轮",在不破坏现有 ContextSource 抽象的前提下,把 prompt 结构改成 cache 友好版,并把命中数据接到 token 看板。

### 1 · 重新分层(三段式)

把现有 7 条 source 按"稳定性"重新归类,渲染顺序改为:

```
─── 稳定前缀(尽量大,争取被全量 cache) ───
[system]   system-prompt                              (静态)
[system]   persona-stable    · 全量 confirmed 列表,固定字典序  (慢变)
[system?]  session-topic     · 仅在话题"已锁定"时进稳定段       (慢变)
[tools]    完整 tools schema,固定字典序                       (静态)

─── 长期记忆缓冲区(中等稳定,会增长但不"抖动")───
[messages] chat-history                                 (累积)

─── 当前轮动态(每轮重算,不参与 cache)───
[system]   ephemeral-context · 合并 working-memory / relevant-memory /
           reference-tag / session-topic(未锁定时)        (高频变)
[user]     当前用户输入
```

关键变化:

- **persona top-K 改"全量 + 固定排序"**:`agentTopK=10 / userTopK=8` 的策略原本是为了控总长度,但用户量级下注入条数远小于上限,top-K 切片纯粹增加 cache miss 概率。改成"全量 + `id` / `createdAt` 字典序",新增 persona 时**只追加**到末尾,不重排已有项——天然 cache 增量。
- **session-topic 二态化**:话题已锁定(用户 `/topic xxx` 显式设置)→ 进稳定前缀;辅助 LLM 自动识别且**未锁定**→ 进 ephemeral 段。锁定状态由 memory store 一个 boolean 区分。
- **新增 `EphemeralContextSource`**:把所有"高频变"的注入合并到一条 `[system]` message,并放到 `chat-history` **之后**、`user input` **之前**——这条消息是"前缀的最后一公里",它的变化不会影响前面已经 cache 住的段。
- **chat-history 位置上提**:从最末尾移到稳定段后,确保历史本身可以 cache 命中。每轮新增的 user/assistant 消息只是"在末尾追加",前缀完全不变。

> 实施层面:`Agent.composeMessages` 的拼装规则改为按**新增的 `cacheTier` 字段**(`'stable' | 'history' | 'ephemeral'`)排序,priority 仅在同一 tier 内决定相对顺序。这样旧的 priority 数字保留作为兼容入口,新代码逐步迁移到 cacheTier。

### 2 · 观测先行(必须先做,作为优化的标尺)

不能等改完才看效果——**先把命中率埋点接上**,以便每次结构调整都能量化。

- **Provider 层**:`extractUsage` 扩展为读取 DeepSeek 的 `prompt_cache_hit_tokens` / `prompt_cache_miss_tokens`、Qwen 的 `cached_tokens`。
- **`ChatChunk.usage` 类型**:新增 `cacheHitTokens?: number` / `cacheMissTokens?: number`。
- **看板对接**:把命中字段汇入 `ideas-014`(Token 用量看板)的聚合维度,在 UI 里多一条"cache 命中率"折线。
- **日志级别**:debug 日志按轮打印 `cache hit / miss / total prompt`,本地排查方便。

### 3 · 各 Provider 的 cache 开关

- **DeepSeek**:无需任何参数,默认开启,只是"读不读 usage 字段"的问题——观测做完即可。
- **Qwen**:Bailian 部分模型(如 `qwen-max-context-cache-*`)需在请求里显式 `enable_thinking` 同级加 `enable_cache: true`,或在 `extra_body` 注入。`QwenProvider.getRequestBodyExtras` 里加一个开关,**默认开**,允许配置页关闭。
- **未来 Provider**(OpenAI / Anthropic / Moonshot):接口层留出 `cacheControl` 字段,Anthropic 风格(显式 `cache_control: { type: 'ephemeral' }` 标记块)与 OpenAI 风格(自动)走同一抽象。

### 4 · Tools schema 稳定化(辅助,小工程)

- 检查 `tools` 数组传入顺序是否在不同会话里稳定——目前 `Agent.tools` 来自 `bootstrap` 装配,顺序应固定,但需要测试断言。
- 给 `ToolDefinition` 的 `parameters` JSON Schema 做 key 字典序排序(渲染层),避免 `JSON.stringify` 因 key 顺序不同导致字节流抖动——这种坑历史上踩过。

## 备注 / 风险

- **核心思想的语义反转**:从"每轮重组以减少 token"→"前缀尽量稳定以让模型自己 cache"。这两个目标在**短前缀场景**下确实矛盾(没 cache 时,长前缀 = 多 token),但在**长会话**(>10 轮)和长 system prompt 场景下,cache 命中比"省那几个 token"重要得多。文档里要把这个权衡写清楚,避免后续 PR 又走回"动态裁剪"的老路。
- **persona 全量化的隐患**:如果用户审核通过的 persona 数量上百,全量注入会让 prompt 变长。当前观察上限是个位数到十几条,远低于阈值;但需要在 `EphemeralContextSource` 旁边留一个"persona 软上限"开关,超出时降级为 top-K。
- **`reference-tag` 进 ephemeral**:用户在引用 chip 上花了精力,把它放到 ephemeral 段意味着每轮都要重算——但本来引用就是"当前轮的"语义,这是合理的归位,不算回归。
- **DeepSeek cache 命中**:64 token 最小单元 + 12 小时 TTL,实测下来"短 system + 短 history"的会话可能依然 miss——所以这个 idea 对长 system prompt(我们的 system + persona 段加起来通常 > 1k token)价值最大,对纯短问答场景收益有限。
- **与 ideas-014(Token 看板)、ideas-015(AgentRuntimeConfig)联动**:命中字段写哪张表、cache 开关由 AgentRuntimeConfig 集中管理,先在本条目里"先观测后控制",落到 014/015 时再迁移。
- **不在范围**:Anthropic 的显式 `cache_control` 块标记策略——等 Anthropic Provider 真正接入(`backlog` B-001 同类项)再单开 idea。

# 按模型能力动态设置上下文上限

- **ID**: ideas-019
- **状态**: proposed
- **优先级**: P1
- **来源**: 用户反馈 · 自查 `packages/shared/src/config/defaults.ts#maxContextChars` / `packages/ui/src/features/options/{OptionsForm.tsx,tabs/BasicTab.tsx}` / `packages/provider/src/{qwen,deepseek}/config.ts#*_MODEL_CAPABILITIES`
- **创建日期**: 2026-05-27

## 背景

`ChatSettings.maxContextChars` 是发送给 LLM 的"携带上下文字符上限",当前实现状况:

| 项 | 现值 | 来源 |
| --- | --- | --- |
| 默认值 | **8 000 字符**(中文 ≈ 2.5K token) | `packages/shared/src/config/defaults.ts#L104` |
| UI 滑杆下限 | 1 000 | `BasicTab.tsx#L381` |
| UI 滑杆上限 | **32 000 字符**(中文 ≈ 10K token) | `OptionsForm.tsx#L122` zod schema + `BasicTab.tsx#L382` |
| 实际消费 | `chat-history.ts#truncateHistory(maxChars)` 按字符从尾部留 | 仅作用于 chat history,不含 system / persona / tool result |

而 `packages/provider/src/{qwen,deepseek}/config.ts` 里**已经存在** `contextWindow` 能力表:

```ts
// qwen
qwen3-max:        262_144   // 256K
qwen-plus:        131_072   // 128K
qwen-turbo:        32_768   // 32K(默认兜底)

// deepseek(v0.6.0-beta.2)
deepseek-v4-pro:   1_000_000  // 1M
deepseek-v4-flash: 1_000_000  // 1M
```

**矛盾**:DeepSeek v4 用户的 context window 是 **1 000 000** token,而我们的滑杆上限是 **32 000 字符**(≈ 10K token)——**用户能用的容量被产品端硬卡到能力的 1% 不到**。即使用 qwen3-max(256K),也只能用上 13%。这把"长上下文模型"的核心卖点直接挡死,且默认 8 000 字符在长文档阅读 / 多轮深度对话场景下经常导致历史被截断。

更深一层的问题:`maxContextChars` 是**纯字符**预算,跟模型实际计费的 token 不是同一个单位,且**不区分系统消息 / 历史 / 当前轮内容**——是一个粗糙的"全局历史尾部裁剪器"。当模型从 32K 提升到 1M,这个粗糙度越发刺眼。

---

## 提案

把"用户手动拉滑杆 + 字符数估算"改成"**Provider 自报能力 → Agent 按比例分桶 → UI 仅作为可选覆盖**"。

### 1 · 定义统一的能力模型

`LLMProvider` 接口新增一个**同步**方法 `getCapability()`,返回:

```ts
export interface LLMCapability {
  /** 模型自身的 context window(token 数) */
  contextWindow: number;
  /** 输出上限(token);DeepSeek v4 = 384K,Qwen 系列各异 */
  maxOutputTokens: number;
  /** 是否支持 reasoning(qwen3 / deepseek-v4 thinking 模式 / GPT o1) */
  supportsReasoning: boolean;
  /** 是否原生支持 tool call(影响 last-turn 兜底策略) */
  supportsTools: boolean;
  /** 来源:'static'(本地能力表)/ 'remote'(模型 API)/ 'fallback'(未知模型保底) */
  source: 'static' | 'remote' | 'fallback';
}
```

实现:

- **Qwen / DeepSeek**:已有 `getQwenCapability(model)` / `getDeepSeekCapability(model)`——直接包一层即可,`source = 'static'`,未识别模型走 `*_DEFAULT_CAPABILITY`(`source = 'fallback'`)。
- **后续 OpenAI / Anthropic**:同样静态表。
- **远程动态**(可选,后期):若 Provider 在 `listModels` 里返回了 `meta.context_length`(如 OpenAI 的 `/v1/models/{id}` 详情),直接 merge 上来,覆盖静态表,`source = 'remote'`。

### 2 · 把 `maxContextChars` 升级为"分桶预算"

`ChatSettings` 引入新字段(同时保留旧字段作为 v0.x 兼容,在 bootstrap 里映射):

```ts
export interface ChatSettings {
  systemPrompt: string;
  maxTurns: number;

  /**
   * 上下文预算策略
   *   - 'auto'   : 由 Provider capability 自动决定(推荐,默认)
   *   - 'manual' : 用户手动指定 totalTokenBudget
   * 旧 maxContextChars 在升级时映射成 manual + 同等 token 估算
   */
  contextBudgetMode: 'auto' | 'manual';

  /**
   * manual 模式下的总 token 预算;auto 模式下忽略,运行时由 capability 决定
   * 上限不再写死 32K——由 Provider capability 动态约束(滑杆动态上限)
   */
  totalTokenBudget?: number;

  /** auto 模式下的安全余量(0-1),默认 0.3 = 留 30% 给输出 + 突发膨胀 */
  autoModeSafetyMargin?: number;
}
```

**Auto 模式下的预算分配**(粗版,后续可再细化):

```
total      = capability.contextWindow * (1 - safetyMargin)
            - capability.maxOutputTokens         // 给 completion 留位置
prefix     = systemPrompt + persona + tools schema    (不裁,超了报错)
ephemeral  = working-memory + relevant-memory + ref   (按 priority 截到 ≤ 15% total)
history    = total - prefix - ephemeral - currentTurn  (历史吃剩下的)
```

历史裁剪器 `truncateHistory` 的入参从 `maxChars` 升级到 `historyTokenBudget`,内部按"粗略 token 估算"(中文 ≈ 1.7 char/token,英文 ≈ 4 char/token,可配)从尾部保留——精确 tokenizer 留给后期(`tiktoken` 体积大,延后接)。

### 3 · UI 改造

**Basic Tab "对话行为" 卡片**重做:

- **上下文预算**:Radio `auto / manual` 二选一,默认 auto。
- **auto 模式**:展示当前 Provider 的 `contextWindow`(灰色不可编辑),旁边显示"安全余量"小滑杆(默认 30%);标签写"由模型能力自动决定 · 当前 1 000 000 tokens(DeepSeek v4 Pro)"。
- **manual 模式**:滑杆 `min=1000`、**`max=capability.contextWindow * 0.95`** 动态约束;单位从字符改 token,提示文案"由模型能力上限决定;切换模型时本值会自动夹回新模型的上限"。
- **切换 Provider 时的兜底**:bootstrap 检测到 `totalTokenBudget > newCapability.contextWindow * 0.95` → 自动夹回上限 + 在配置页顶部弹一条 banner"已自动调整到新模型的上下文上限,你可以重新设置"。

### 4 · 旧版字符模式的迁移

- `maxContextChars` 字段保留在 schema 里至少 2 个 minor 版本,标记 `@deprecated`。
- bootstrap 装配时:若新字段不存在但旧字段存在 → 按 `manual` + `totalTokenBudget = round(maxContextChars / 1.7)` 自动迁移,并在日志写 `migrated maxContextChars=8000 → totalTokenBudget=4700 (manual)`。
- CHANGELOG 单独写一条 breaking 提示(本项目尚未公开 release,迁移路径不需要太重)。

## 验收标准

- [ ] **能力声明**:`LLMProvider.getCapability()` 在 Qwen / DeepSeek 落地;`PROVIDER_REGISTRY` 表暴露默认能力,`listModels` 把 `capability` 字段传到 UI。
- [ ] **schema 升级**:`ChatSettings` 新增三字段;bootstrap 实现旧字段迁移并加单测。
- [ ] **历史裁剪器**:`truncateHistory` 改吃 `historyTokenBudget`,粗 tokenizer 走 char/ratio 路线,中英文比率可在 `shared/config` 调整。

  - DeepSeek v4 Pro(1M) + safetyMargin=0.3 + 长 system prompt(2K)→ history budget ≈ 数十万 token

- [ ] **UI 改造**:Basic Tab 落地 `auto/manual` Radio + 动态滑杆;切换 Provider 时滑杆 max 跟着变。
- [ ] **滑杆 max 不再硬编码 32000**:删掉 `OptionsForm.tsx#L122` 的 `.max(32000)`,改由动态 capability 约束。
- [ ] **质量门禁**:`pnpm typecheck` / `pnpm lint` / `pnpm test` 全绿。

## 备注 / 风险

- **跟 ideas-018(Prompt Caching)的协同**:更大的 history budget 意味着 chat-history 更长 → cache 命中收益更显著。两条 idea 应该一起落地,**先 018(改前缀稳定性)再 019(放开预算)**——预算放开后冗长前缀的 token 浪费才不会立刻被放大。
- **跟 ideas-014(Token 看板)的协同**:看板里加一列"本会话 prompt token / 当前模型 contextWindow"百分比柱形,让用户直观看到"还离上限多远"。
- **跟 ideas-015(集中化 AgentRuntimeConfig)的协同**:`getCapability()` 的结果天然是 AgentRuntimeConfig 的一项;最终方案里 `maxContextChars / totalTokenBudget` 只是 runtime config 的一个字段,本 idea 的 UI 可视为 015 的一个具体落地切片。
- **粗 tokenizer 的精度问题**:char/ratio 在多语言混杂(代码 + 中文 + emoji)的真实页面里偏差可达 ±30%。短期接受这个误差,等真有用户反馈"被截了"再上 `tiktoken-rs` / wasm 版精确 tokenizer——它压扩展体积接近 1MB,需要权衡。
- **不在范围**:
  - 输出端 `max_tokens` 自适应(可单独成 idea,与 reasoning_effort 联动)
  - 摘要式压缩(summarize-then-include)长 context 仍需要的高级裁剪策略——延后到 ROADMAP "高级 context 压缩"
  - 模型在线探测 contextWindow(各家 `/v1/models` 详情字段不一致;先做静态表 + Provider 自报)
- **对默认值的判断**:把 default 改成 `auto + safetyMargin=0.3` 后,新用户在 DeepSeek 上等于"几乎不限制"——这是好事,但需要在文档(`docs/CHANGELOG.md` + Options 页提示)里写清,避免用户疑惑"为什么变了"。

## §1 · 版本概览

### 两版主题

- **v0.4.0 · 可见且可按时间检索的记忆系统**:把记忆层从"能用但不透明"的黑盒,变成用户可以**浏览、检索、审核**的"时序自传"。
- **v0.5.0 · 统一记忆 · Offscreen Document 架构**:把 IndexedDB 从 content-script origin 搬到扩展 origin 下的 Offscreen Document,**所有域名共用同一套 DB**。

### 为什么一起发?

v0.4.0 计划独立发版。真机验收时,记忆浏览器 Tab 装在 Options 页(扩展 origin),打开一看——**空的**。深入一查:sidebar 跑在宿主页面 origin(bilibili/github/...),每个域名一份独立的 IDB,Options 页看到的是**第四份**空 IDB。

这是架构级问题,不是 v0.4.0 的 bug,是 v0.2 起就埋着的隐患第一次被**新功能放大**。如果不 fix,v0.4.0 的记忆浏览器就是残疾品。所以 v0.4.0 的 tag 暂停,切到 v0.5.0 上下文连续地把架构修掉,两个版本合并成一次 release 打 tag。

### 质量指标

- 测试用例:**315 → 533**(+218),主要增量来自 `list_recent_visits` tool / 双主体 Persona / `remote-store.test.ts` 22×3 路径覆盖 / MemoryBrowserTab
- 提交数:发版期共 **18 个 commit**(不含 merge,含则 19)——v0.4.0 实占 8 个(docs + memory 契约 + 需求 1~5 + TODO 搭车修),v0.5.0 实占 10 个(docs + PR-1 + PR-2 + PR-3 + 3 个 hotfix + merge commit)
- 所有 typecheck / lint / test 全绿后打 tag `v0.5.0`

## §2 · v0.4.0 的关键设计

### §2.1 Persona 双主体 · 从"指令 vs 事实"到"定义 agent 还是定义 user"

v0.4.0 的第一条需求是给 `PersonaRecord` 加 `subject: 'agent' | 'user'` 字段。表面看,这是 v0.2.2 "Persona 全部当 Agent 长期指令"那次反转的延续——但真正让这次设计立得住的,是**分类维度本身的纠正**。

**最初 PM 的心智模型**:"指令 vs 画像"。agent 侧存"你该怎么做"(指令),user 侧存"用户是什么样"(画像/事实)。`remember_persona` tool 的 description 初稿就是按这个写的。

**用户一句话点破**:这个归类是错的。本质不是"指令还是事实",而是"**这条信息在定义谁**"。

- "你叫小瑾" —— 是在**定义 agent 的身份**,不是在给 agent 下指令
- "你回答要简洁" —— 是在**定义 agent 的行为方式**
- "我是前端工程师" —— 是在**定义 user 的身份**
- "我偏好 TypeScript" —— 是在**定义 user 的偏好**

按"指令 vs 事实"分类,agent 侧的"你叫小瑾"会很尴尬——这既不是纯指令(没让它做什么),也不是对 user 的事实。按"定义谁"分类,一切自然归位:**agent 侧有 agent 的身份/性格/行为方式,user 侧有 user 的身份/背景/偏好**。

PM 按新维度**全量修正了措辞**,至少 10 处:tool description、反思 Job prompt、`PersonaSource` 注入段落的 section title、banner 徽章文案("[关于你]" / "[关于用户]")、反思 Runner 的 parser 返回值。

**教训:产品语义要讲精准**。"指令 vs 事实"听起来合理,但模型在第二人称语境下("你叫...")就会混淆——这既像指令又像 identity。"定义谁"是更本质的维度,agent 侧也有大量"身份/性格"内容(不全是指令),user 侧也有影响 agent 表达的内容(不全是事实)。

**这不是第三次反转**。v0.2.1 → v0.2.2 是"user 画像 → agent 指令"的**替换**;v0.4.0 是在 v0.2.2 之上**叠加一个视角维度**。同一张表,两类 subject 共存,不推翻 v0.2.2 的转译规则——`remember_persona` 只是新增了 subject 必填参数,把两类定义分开存、分段注入。

### §2.2 Chronological Index · 摇摆两次后回到原方案

时间维记忆检索("今天看了什么"/"本周读了啥")在 v0.2.5 就登记为"意图驱动召回"的第二档,这次终于要落地。PM 反复摇摆了两次。

**第一轮**:PM 想法——不改 `recall_memory`,新增独立 tool `list_recent_visits`。两个 tool 分工清楚:语义召回走 `recall_memory`(扩 `timeRange/domain` 做窗内过滤),时间维列清单走 `list_recent_visits`。

**第二轮**:用户反对——"tool 越多 LLM 越混乱"。改成**单 tool 双 mode**:`recall_memory({ mode: 'semantic' | 'time' })`。PM 按这个改完了 schema 和 runner。

**第三轮**:用户又改回去——回到两个 tool。并明确了**真正的理由**:"单 tool 双 mode 反而让 LLM 理解负担更大。它得先判断自己该传什么参数,再看哪些必填,再关心错传时的降级。两个 tool 各自参数稳定必填,description 能各自写清意图,LLM 在选型阶段就已经做完决策了。"

最终定稿就是两个 tool:
- `recall_memory`(`query` 必填,可选 `timeRange/startTs/endTs/domain/articleId/limit`)
- `list_recent_visits`(`timeRange` 必填,可选 `startTs/endTs/domain/limit`)

同时在 `RelevantMemorySource` 内做"**系统侧隐式路由**":粗判命中时间维关键词("今天/本周/昨天")→ 跳过向量,直接按 `timeRange` 拉 `visit_summary` 清单注入。用户/LLM 哪条路径都走得通,底层共享 `packages/agent/src/context/time-query.ts` 里的 `detectTimeScopedMetaQuery` + `resolveTimeRange`。

**顺带清理**:把 v0.2 就埋下的 `mode: 'auto' | 'explicit'` 枚举、`time_query_unsupported` 降级分支全删。这两个是 v0.2.1 召回链埋的"双模式入口",实际使用里 `explicit` 只被 `/recall` 和 tool 调用走,`auto` 永远是 `RelevantMemorySource` 自己走——作为 tool 参数存在毫无意义,纯粹制造认知负担。

**教训:反复摇摆很正常,但每次改动都要把历史决策的原因讲清楚**。最后定两 tool 的理由不是"tool 多好",而是**"每个 tool 的参数稳定必填"让 LLM 的选型决策前置**——这和"单 tool 双 mode 让 LLM 判断自己传啥"是两种负担模型。如果没把这个理由写进 CHANGELOG 和需求文档,三个月后回来看代码很可能又想合并回去。

### §2.3 记忆浏览器 Tab · 一次"副作用"暴露架构漏洞

最初 PM 方案很简单:Options 页新增第五个 Tab `MemoryBrowserTab`,直接 `new DexieMemoryStore()` 读 `episodes_visit_summary`,按日期/域名分组。顺手给 `MemoryStore` 加 `deleteRecord(id)` 必填方法,支持单条删除。

代码写完,typecheck 0 error,单测全绿。真机打开 Options Tab——**空的**。

DevTools 一查:每个宿主域名(bilibili/github)下都有一份 `doc-assistant` IDB,**Options 页自己也有一份(空的)**。这是 v0.2 起就存在的"按 origin 隔离"问题第一次被**跨域查看场景**放大。

之前 v0.2.1 → v0.4.0 期间,一直按"每个宿主域名各自一套记忆"运行,没人把它当问题——因为没有跨域查看的场景。sidebar 在宿主域名上聊天,召回自己同域名的记忆,用户感觉不到问题。直到 v0.4.0 的**记忆浏览器 Tab**把数据拿到 Options origin 来显示,这份 IDB 的**归属视角**才第一次被暴露。

**教训:架构漏洞的"副作用"往往通过新功能被第一次放大**。v0.2 ~ v0.4 的日常测试完全无法发现这个问题,它必须要"**把数据拿到另一个 origin 去显示**"这个具体场景才会显现。这也直接触发了 v0.5.0 的架构级重构。

**此外顺带做掉的**:`/forget` 的删除能力没有做新 slash 命令(避免新增命令制造认知负担),而是以**记忆浏览器 UI 按钮**形式落地。`MemoryStore.deleteRecord(id)` 变成接口必填方法,`NullMemoryStore` 给 no-op 实现,`DexieMemoryStore` 幂等删除(不存在不抛)。

### §2.4 话题漂移关键词触发 · 用 0.5h hack 替代 2-3h 完整方案

需求列表里 PM 最初想做"**完整版话题漂移主动检测**":相似度阈值 + 每轮 aux 判漂移。工时估算 2-3h。

PM 自己否了这个方案。

原因:**真机数据不足以调参**。相似度阈值要跑多少真实场景才能确定?aux 判漂移的 prompt 要调多少轮才稳定?我们目前还没有积累足够多的真实对话数据来支撑这些决策——现在调出来的参数可能是**给单元测试调的**,不是给真实用户调的。

换成 0.5h 的 **hack**:`shouldIdentify(count, { latestUserInput })`,匹配 `TOPIC_DRIFT_PATTERNS`(正则,中英双语),命中"换个话题 / 换话题 / 说点别的 / let's switch / new topic / ..."即立即重识别。完整版留 v0.5+。

**教训**:
1. **不要在真空里调参**。相似度阈值/aux prompt 这类"需要真实分布才能决定"的参数,真机数据不足时先用 hack 覆盖高置信场景,给后续真机积累留空间。
2. **不要把 v0.5+ 的工作拉进本期**。关键词 hack 够覆盖"用户明确说换话题"的高置信场景,已经能解决 80% 的"话题滞后"体验问题。完整版的 ROI 要等 hack 跑一段真机数据再评估。

### §2.5 host_permissions 放开 · 产品原则要提前澄清

v0.1.1 §4 登记过"manifest `host_permissions` 只覆盖千问两个域"的坑,v0.3 Provider 抽象后用户自配 OpenAI / 自托管千问的 baseURL 直接被 CORS 拦截。v0.4 一次解决:`host_permissions: ["<all_urls>"]` + PRIVACY/CWS 英文 justification + CHANGELOG 写清 breaking。

**教训:产品原则要提前澄清,PM 不要自作主张加兜底**。PM 起初想过提示词限制、`optional_host_permissions` 动态申请、脱敏开关——**用户明确拒绝,理由是"各厂商已有脱敏,不做重复工作,用户填 baseURL 即主体意志明确"**。每多一层兜底都是未来维护成本,只在产品原则模糊时才做保护性设计;原则明确时 PM 自加兜底反而是噪声。

### §2.6 搭车修 TODO 不标记 · v0.2.x 老 bug

真机验收 v0.4.0 时发现 v0.2.1 起就有的老 bug:LLM 完成 TODO 对话后不主动调 `complete_todo` tool,WorkingMemory 卡片里的 TODO 永远未完成。根因:system prompt 的弱引导("完成后及时 `complete_todo`")模型不主动配合;注入格式也没带 id,LLM 即使想调也不知道参数。

修复两手:强化 system prompt 用"**必须**"措辞 + 注入时每条 TODO 带 `{id=...}` 后缀让参数有据可依。

**教训:LLM 的弱引导容易被忽略,越强越好**。柔性提示在模型注意力下极易失效,关键动作要用"必须"+ 把必填参数的值显式注入上下文才能形成闭环。本期真机跑一段未观测到过度调用或误删的副作用。

## §3 · v0.5.0 的架构决策

### §3.1 问题的发现过程

v0.4.0 真机三连问:
1. "记忆浏览器 Tab 是空的" —— Options origin 一份独立 IDB
2. "WorkingMemory 里的 TODO 永远不标记" —— §2.6 已修
3. "Persona 列表是空的" —— 同 1,审核 banner 在 sidebar 看起来正常,去 Options 想批量管理就是空

根因定位非常快——**Chrome 扩展 IndexedDB 按 origin 隔离**,sidebar 跑在宿主页面 origin,bilibili 一套、github 一套、Options 页 `chrome-extension://<id>` 又一套。四份独立 IDB,**拼起来才是完整记忆**,任何单一视图都残缺。

**最值得自责的是**:`docs/TROUBLESHOOTING.md §8` **明确警告过**这件事。那条 2021 年 v0.2.1 登记的"SW 与 sidebar IndexedDB 同源隔离风险"写得清清楚楚:SW(扩展 origin)不能访问 sidebar(宿主 origin)的 IDB。当时为了反思 Job 能跑,用了"SW 只唤醒、sidebar 执行"的绕路方案规避。

做 v0.4.0 记忆浏览器 Tab 时,**漏看了这条历史**。Options 页和 sidebar 的 origin 关系和 SW 与 sidebar 是**同一类问题的不同具象**,但 PM 当时没把它识别出来,以为"Options 读 Dexie 天经地义"。

**教训:历史文档是真金白银,每次架构设计要当第一检查项**。TROUBLESHOOTING.md 不是摆设,里面每一条都是代码代价换来的。下次做架构改动,**强制把 TROUBLESHOOTING 的所有历史条目刷一遍**,识别"我本次涉及的假设是否有历史条目已经反驳过"。这是 PM 自己的操作失误,不应该让下一代 PM 再交一次学费。

### §3.2 三个方案对比

定位问题后,PM 和 Plan agent 讨论了三个候选方案。

**方案 A · SW 持 DB + 消息桥**

> 思路:让 Service Worker 自己 `new DexieMemoryStore()`,sidebar/Options 都通过消息代理读写。

否决:**MV3 SW 有 30 秒空闲挂起机制**。Chrome 为省内存会把无事可做的 SW 暂停,流式 tool-calling 这种长链路会被中断,每次唤醒都要重建 DB 连接、损失请求状态。反思 Job 要跑 30 秒以上更是常态,一定会被挂掉。

**方案 B · iframe 方案**(B1 content script 嵌扩展 origin iframe / B2 Chrome 114+ Side Panel API)

否决:**UX 回归成本过高**。Lexical / 划词 / Shadow DOM 这三条能跑到今天都付过代价(TROUBLESHOOTING §1、§6、§7),iframe 里全部要重写。用户还提醒:"**划词在 iframe 下有问题**"(跨 frame selection API 不共享,划词引用功能直接报废)。

**方案 C · Memory Remote Adapter**(最终选)

这个方案是**用户直接提出的**。当 PM 和 Plan agent 陷入"sidebar 迁到 iframe"的思考怪圈时,用户一句反问破局:

> "为什么不能分两层?把写入替换成一个个原子操作,通过事件形式在插件层写入。"

这句话把思路从方案 B 直接拉到方案 C:`MemoryStore` 接口天然是 **async 原子操作**(22 个方法),非常适合 RPC 化;sidebar 不持 DB,只持 `RemoteMemoryStore` 代理,所有方法转成 `chrome.runtime.sendMessage({ type:'MEMORY_RPC_REQUEST', method, args, rpcId })`;真实 DB 搬到扩展 origin 下的宿主(SW 或 Offscreen),sidebar 和 Options 走同一条 RPC 通道;**UX 零改动**,只换数据层不换 UI 层。

**教训:用户比 PM 更懂自己的架构意图**。PM 和 Plan agent 一开始被"sidebar 要迁到 iframe"这个方向锁死,用户一句"为什么不能分两层"就直接破局。PM 要学会**听用户问"为什么不能"的反问**——这种反问往往意味着用户已经看到了一个更清洁的拆分,而你陷在了一个错误的默认值里。

### §3.3 Offscreen vs SW 直持 DB

方案 C 定了,还有个子问题:**把 DB 搬到扩展 origin 下的哪里**?

两个候选:
- **选 A · SW 持 DB**:SW 自己 `new DexieMemoryStore()`,`onMessage` 处理 RPC
- **选 B · Offscreen Document**:SW 只负责保活路由,DB 放在 offscreen(一个常驻的隐藏 DOM 文档)

选 A 貌似"简单",但会重演方案 A 的问题:SW 30 秒挂起会导致 DB 连接重建、反思 Job 无处安放。选 B 的代价是多一个 `offscreen.html` + manifest 加 `"offscreen"` 权限,但收益一箩筐:

- **DB 连接常驻**,不跟随 SW 挂起重建
- **反思 Job 可以跑在 offscreen 里**(长任务不怕被挂)
- **彻底解除 §8 的 SW 唤醒 sidebar 绕路**——SW 只做 alarm 监听 + 转发 `REFLECTION_TICK` 到 offscreen,反思 Job 不再依赖 sidebar 在线

三者合一,Offscreen 是**一次做对**的选择。SW 直持 DB 看似省事,但会留下"SW 挂起 → 反思 Job 怎么办"这个技术债,未来还得补。

**教训:一次做对,不要为了省事留技术债**。多一个 `offscreen.html` 文件 + `"offscreen"` 权限的工作量,比"先用 SW 扛着、未来再迁"的隐性成本低一个量级。

### §3.4 实施路径 · 3 PR 各自闭合

整体 ~10h 工作量,按 3 个 PR 拆:

- **PR-1 · RemoteMemoryStore + Offscreen 骨架**(~5h):定义 22 条 RPC、实现 `RemoteMemoryStore`、拉起 offscreen、sidebar/Options bootstrap 切换;`remote-store.test.ts` 覆盖 22×3 路径。
- **PR-2 · 反思 Job 迁移**(~4h):`ReflectionRunner/Scheduler` 搬到 offscreen、SW alarm 改为转发 `REFLECTION_TICK`、sidebar 删除 `REFLECTION_SCAN_TICK` 监听、加 `PAGE_VISIT_ENDED` 消息。
- **PR-3 · 清理 + 文档 + 发布**(~1h):TROUBLESHOOTING §8 打"v0.5.0 已解除"徽章、README 写 Chrome 109+ 要求、CHANGELOG、tag。

每个 PR 自闭合——PR-1 合并后系统可用(反思 Job 仍跑 sidebar),PR-2 合并后反思 Job 已迁移但老消息枚举暂存,PR-3 清尾。

## §4 · v0.5.0 真机踩的 3 个架构坑

这 3 个坑都是**单测覆盖不到**的——需要真机在真实 Chrome 扩展环境里跑 offscreen + SW + sidebar 三个进程才会暴露。

### §4.1 坑 1 · 冷启动窗口 RPC 丢失(commit `0adc518`)

**现象**:sidebar 启动后立刻发 RPC,offscreen 还没就绪,返回 undefined → Console 报 `"unexpected RPC response shape"`,首次召回 / Persona 列表全部拿不到数据。

**根因**:SW 的 `ensureOffscreenAlive()` 是异步的(内部要调 `chrome.offscreen.createDocument`,这个 Promise 得等 Chrome 内核真正拉起文档)。sidebar 启动时机**早于** offscreen 就绪。第一条 RPC 打过来时,SW 才刚开始拉 offscreen——消息被路由但没有消费者。

**修复**:在 `packages/memory/src/remote/defaultChromeRuntimeTransport` 里加**线性退避重试**:`DEFAULT_RPC_RETRY_MAX=3`、`DEFAULT_RPC_RETRY_BASE_MS=150`,即 150 / 300 / 450ms 共 ~900ms。超过后抛 `MemoryRpcTimeoutError`。

```ts
// 伪代码
async send(req) {
  for (const delay of [0, 150, 300, 450]) {
    if (delay) await sleep(delay);
    const resp = await chrome.runtime.sendMessage(req);
    if (isValidResponse(resp)) return resp;
  }
  throw new MemoryRpcTimeoutError();
}
```

**教训:任何跨进程的"先启动消费者,再启动生产者"的时序依赖都要用重试兜底,不能依赖"总会赶上"的假设**。消费者(offscreen)在另一个进程异步拉起,生产者(sidebar)在用户点开 sidebar 的瞬间就开始发消息,这种 race 永远存在,**代码必须自己处理**。

### §4.2 坑 2 · `IDB_PERSISTENCE` 不是合法 reason(commit `fd8ee30`)

**现象**:
- `chrome://extensions` 找不到 offscreen 检查视图(正常情况下应该能看到 offscreen 的 DevTools 入口)
- SW 控制台报 enum 错误
- 所有 RPC 都打不通

**根因**:PR-1 里 `chrome.offscreen.createDocument({ reasons: ['IDB_PERSISTENCE' as chrome.offscreen.Reason] })`——`IDB_PERSISTENCE` 是 PM 想当然取的名字(语义上最贴近"持久化 IndexedDB"),TS 类型不认识,PR-1 用了 `as chrome.offscreen.Reason` 硬 cast 绕过。

Chrome 官方 Reason 白名单里**没有这个值**:`TESTING / AUDIO_PLAYBACK / IFRAME_SCRIPTING / DOM_SCRAPING / BLOBS / DOM_PARSER / USER_MEDIA / DISPLAY_MEDIA / WEB_RTC / CLIPBOARD / LOCAL_STORAGE / WORKERS / BATTERY_STATUS / MATCH_MEDIA / GEOLOCATION`。传非法值,`createDocument` 抛错 → offscreen 永远起不来 → 所有 RPC 拿不到 response。

**修复**:
- 改用 `LOCAL_STORAGE`(IndexedDB 与 localStorage 同属 storage partition,Chrome 官方对 IDB 宿主没专用 reason,社区通行写法即 `LOCAL_STORAGE`;不用 `BLOBS`——语义是"保存用户生成 Blob",不符)
- 新增 `verifyOffscreenAlive('sw-boot')` 自检(`apps/extension/src/background/memory-handler.ts:93`,由 `apps/extension/src/background/index.ts:28` 在 SW 启动时 `void verifyOffscreenAlive('sw-boot')` 调一次):内部先 `ensureOffscreenAlive()`(幂等 createDocument),再 `chrome.offscreen.hasDocument()` 确认,返回 boolean 并打日志 `[self-check:sw-boot] offscreen document 可用性: ok/failed`——把拉起失败在 SW 启动阶段暴露,不用等到第一条业务 RPC 才发现

**教训:`as` 硬 cast 是魔鬼**。TS 类型不知道这个值合法,说明它**可能就是不合法**。硬 cast 把编译期错误推迟到运行时,极难排查——本次 reason 错 10 分钟就能改,但架构复杂点的地方一旦 cast 藏了假设,debug 成本会指数上升。**遇到 `as` 想写的时候,先去查官方 API 文档确认值合法,再决定要不要 cast**。

### §4.3 坑 3 · offscreen 不能用 `chrome.storage`(commit `df62b8b`)

**现象**:解决前两个坑后,所有 RPC 都返回 `[storage] chrome.storage.local is not available`。记忆浏览器空、召回空、反思 Job 不跑。

**根因**:Chrome 官方明确 "offscreen document **只支持 `chrome.runtime` API**",不能直接用 `chrome.storage`。PR-1 的 offscreen bootstrap 在顶层直接 `createTypedStorage()` 读 Provider 配置 → 整个 runtime 构造**失败**,导致所有 RPC handler 都在一个半死的 runtime 里跑。

这个限制**完全反直觉**——offscreen 看起来是一个完整 DOM 环境(有 `document`、`window`、可以 `new Dexie()`),但它的 `chrome.*` API 白名单非常严格。

**修复**:
- offscreen 想读 storage 时,**通过消息让 SW 代读**:
  - `OFFSCREEN_STORAGE_READ_REQUEST` · offscreen → SW
  - `OFFSCREEN_STORAGE_READ_RESPONSE` · SW → offscreen
- offscreen bootstrap 改为:启动 → 发 storage read 请求拿 Provider 配置 → 配置到齐后再构造 `QwenEmbeddingProvider / AuxLLM / ReflectionRunner / ReflectionScheduler`

**教训:Chrome MV3 的各种限制是反直觉的**。看起来 offscreen 是完整 DOM 环境,实际 API 白名单很严格——`chrome.storage` 不能用,`chrome.alarms` 不能注册监听,`chrome.tabs` 不能直接操作等等。**每次设计前必查官方文档的"API 支持性"表**,不要凭"应该能用"的直觉写代码。

## §5 · 本期的关键经验

> 至少 6 条,按重要度排序。

1. **需求的语义要精准**(§2.1)
   Persona "指令 vs 事实" 看似合理,实际让模型在第二人称语境下困惑。"定义 agent 还是定义 user" 才是本质维度。**一个分类维度选错,下游所有话术/schema/UI 文案都得重写**。

2. **历史文档是最好的老师**(§3.1)
   TROUBLESHOOTING §8 已经警告过 "Chrome 扩展 IDB 按 origin 隔离",但 v0.4.0 做记忆浏览器时漏看,直接让架构漏洞暴露。**架构设计前,把 TROUBLESHOOTING 和 DESIGN-HISTORY 的所有历史条目刷一遍,识别"我本次涉及的假设是否已经被反驳过"**。

3. **硬 cast 是魔鬼**(§4.2)
   `'IDB_PERSISTENCE' as chrome.offscreen.Reason` 把编译期错误推迟到运行时,付了三连坑的第一坑。**TS 类型不知道这个值合法,说明它可能就是不合法**。遇到想 `as` 的时候,先去查 API 文档确认值合法,再决定要不要 cast。

4. **用户比 PM 更懂产品**(§3.2)
   Memory Remote Adapter 是用户提出的。PM 和 Plan agent 一度被"sidebar 迁 iframe"锁死,用户一句"为什么不能分两层"直接破局。**用户问"为什么不能"的反问,往往意味着他看到了一个更清洁的拆分,PM 要学会停下来听**。

5. **真机踩坑无可替代**(§4.1 / §4.2 / §4.3)
   3 个架构坑都是单测覆盖不到的——需要 offscreen + SW + sidebar 三进程在真实 Chrome 里跑才会暴露。**扩展类项目发版前必须真机跑核心链路**,单测全绿不代表能上线。

6. **一次做对胜过补丁式修复**(§3.3)
   选 Offscreen 而非 SW 直持 DB,多了一个 `offscreen.html` 和 `"offscreen"` 权限,但彻底解除了 §8 绕路、让反思 Job 有常驻宿主。**"先用 SW 扛着、未来再迁"的隐性成本比"一次做对"高一个量级**,不要为了省事留技术债。

7. **不要在真空里调参**(§2.4)
   话题漂移完整版(相似度阈值 + aux 判漂移)没做,不是因为不重要,是因为**真机数据不足以调参**。先用 0.5h 关键词 hack 覆盖高置信场景,给真机积累留空间,完整版等数据到位再做。

## §6 · 对未来的启示

### 流程 checklist(教训见 §5,不再复述)

**架构设计前**,必做三件:
1. 刷一遍 `docs/TROUBLESHOOTING.md` + 所有 `DESIGN-HISTORY` 条目
2. 识别"**我本次涉及的假设是否已经被反驳过**"
3. Chrome 扩展类改动必查 MV3 官方"API 支持性"表——典型坑:offscreen 不支持 `chrome.storage`(§4.3)、不能注册 `chrome.alarms.onAlarm`(本期设计成 SW 转发)、`chrome.offscreen.Reason` 是严格白名单(§4.2)

**设计文档里写"踩坑清单"章节**(如本期 §4.1~4.3),提前预测,哪怕只是一条"冷启动 race 必须重试兜底"的警句。

**发版前真机跑核心链路**,不是"打开 sidebar 能说话":
- 跨域名记忆连续(bilibili 聊完切 github 能看到 bilibili 的 visit_summary)
- 反思 Job 在关闭 sidebar 后也能跑
- Options 记忆浏览器显示跟 sidebar 一致
- 冷启动场景(Chrome 重启)第一条 RPC 能拿到响应

### 待做事项(ROADMAP P1 之外的延展)

1. **offscreen 可观测性**:目前 offscreen 日志只能从 `chrome://extensions` → "检查视图" 进 DevTools 看。长期需要 Options 或调试 Tab 里加"Offscreen 状态"卡片,显示在线/RPC 队列长度/最近一条 RPC 的 method+elapsed/反思 Job 最近运行时间。

2. **反思 Job 长周期真机压测**:当前只验过分钟级"聊一轮 → 关 tab → 产出 visit_summary"。**超过 1 小时、休眠唤醒、Chrome 重启扩展**等长周期场景未系统压测,offscreen 常驻但 SW 挂起重启时 alarm 转发是否准确触达、反思 Job 是否被不恰当打断,需要真机跑一周看数据。

## §7 · 结语

两个版本合并一次 release 发,总工时 ~23h(v0.4 约 13h + v0.5 约 10h),加上 3 个真机踩坑热修的额外时间,实际消耗更多。最大的收获不是代码,是两个**分类维度的纠正**:

- v0.4.0 · Persona:"指令 vs 事实" → "定义 agent 还是定义 user"
- v0.5.0 · 记忆宿主:"sidebar/iframe 之争" → "数据层与 UI 层分开,RPC 化"

这两次纠正都是**用户直接拍板**的结果。PM 的工作是把用户的直觉翻译成可执行的需求和代码,但也要学会承认:**有时候用户比 PM 更清楚产品该往哪走**。

下一次做大改动,在决策摇摆的瞬间,先停下来看看——用户是不是已经给出答案了,只是自己没听进去?

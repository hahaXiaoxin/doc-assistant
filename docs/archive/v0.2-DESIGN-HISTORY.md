## 一、从"对话工具"到"有记忆的助手"——大版本目标

### 起点（v0.1 MVP）
- 一个能读当前页面、带上下文流式对话的 sidebar
- 无任何记忆：刷新页面 = 从零开始
- 单 Provider（千问），单一 system prompt，无后台任务

### 终点（v0.2.4）
- **四层记忆系统**落地：Persona / SessionTopic / WorkingMemory / Episodic（消息 + visit summary）
- **三套 Provider**：主对话 / 辅助 LLM / Embedding，均支持"复用主 Provider"
- **按需召回**：关键词粗判 → 辅 LLM 精判 → 向量 topK → 邻居消息拼接
- **反思 Job**：PageVisit 结束后异步生成摘要 + embedding + 抽取 Persona 候选
- **意图驱动的记忆召回**：消息自动落库（`episodes_msg`）沉淀数据；刷新后 WorkingMemory 自动注入"我们在做什么"；用户问"上次 XX"走向量召回；用户问"今天/本周 XX"走 Chronological Index（规划中）。**不做 mount 时无脑预热历史**——避免污染上下文
- **话题自动跟随**：SessionTopic 每 4 轮识别；hashchange 切文章会清旧话题
- **跨 visit 消息降级**：切文章后旧对话自动标注为"之前在《...》中"，不污染新文章上下文
- **助手人设化**：主 system prompt 5 段工作守则，tool description 全面精化触发时机

### 哲学转向

**从"工具"到"助手"**：整个 v0.2 的设计都围绕一个转变——
- v0.1 时 Agent 是一个"一次性问答工具"，每轮调用都是独立事务
- v0.2 Agent 是一个"与用户长期合作的助手"，能记住、能承接、会主动维护自己的工作状态

最明显的体现是 v0.2.3 主 system prompt 加入的核心守则：
> 你是一个与用户长期合作的助手，不是一次性的问答机器。自然表达"我们上次聊到..."
> 这种陪伴感，但**不要把内部状态贴在对话里**——不说"根据我的记忆系统..."、
> "让我查一下 WorkingMemory..."。就像一个真人助手不会对你说"让我查一下我的笔记"。

这条哲学影响了多个看似技术性的决策，后面会反复提到。

## 二、记忆系统架构

### 2.1 四层记忆模型

这是整个 v0.2 的骨架。每一层解决一个不同的时间尺度 / 索引维度的记忆问题：

| 层 | 存储表 | 索引 | 生命周期 | 作用 |
|---|---|---|---|---|
| **Persona**（Agent 长期指令） | `persona` | 无（全局） | 永久 | 跨会话 / 跨页面的行为规则 |
| **SessionTopic**（情景记忆） | `session_topics` | `visitId` | 单次访问 | "我们现在在聊什么主题" |
| **WorkingMemory**（工作记忆） | `working_memories` | `canonicalUrl` | 30 天软 TTL | "当前页面 / 任务的 activeGoal + TODO" |
| **Episodic · 消息** | `episodes_msg` | `visitId + canonicalUrl + orderInVisit` | 永久 | 逐条对话消息（反思原料 + 未来 Chronological Index 数据源） |
| **Episodic · visit summary** | `episodes_visit_summary` | 向量 + timestamp + canonicalUrl | 永久 | 反思生成的摘要 + embedding（向量召回） |

**关键设计约束**：
- **每一层都是可选的**：`MemoryStore` 接口里四层的 API 都是 `?` 可选方法；`NullMemoryStore` 对所有方法 no-op。这保证 Agent 层不必知道具体后端（Dexie / 未来的 SQLite / 测试桩）。
- **`remember()` / `recall()` 是永恒接口**：从 v0.1 到 v0.2.4 签名从未改动，所有新能力都以**新增可选方法**的方式追加。
- **分层对 Agent 透明**：Agent 只看到 `ContextSource[]`，不知道每一层来自哪张表。Sources 各自负责从 `MemoryStore` 读自己那层。

### 2.2 为什么是这四层（而不是三层或五层）

这个划分对应 **人类记忆的四个尺度**，不是随便拆的：

- **Persona** = 长期语义记忆（个性、身份）——人类一生都带着的东西
- **SessionTopic** = 工作记忆里的"情景上下文"——你现在在和谁、做什么的即时感知
- **WorkingMemory** = 任务记忆 / 近期计划——类似认知心理学里的"短期记忆 + 执行系统"
- **Episodic** = 事件记忆——具体发生过的事（可追溯、可召回）

这个划分的实用价值是**注入 prompt 时有明确优先级**（见 §3.3）。

### 2.3 ContextSource 优先级（Priority）

这是 v0.2 整个"上下文分层"的核心表现：

```
priority 100  SystemPromptSource        主 prompt（默认行为守则）
priority  80  PageContextSource          当前页面标题 + 摘要
priority  70  ReferenceTagSource         用户划词引用
priority  60  PersonaSource              长期指令（已审核的）
priority  55  SessionTopicSource         当前话题焦点
priority  50  WorkingMemorySource        本页 activeGoal + TODO
priority  40  RelevantMemorySource       按需召回的过往 visit_summary
priority  10  ChatHistorySource          当前对话消息流
```

**Source 的设计原则**：
- 每个 Source 独立 `gather()`，可返回 `null`（意思是"本轮我不贡献段落"）
- 高 priority 的内容在 system prompt 中排在前面
- 所有 Source 都是纯函数：输入 `ctx`（含 `memory` 引用），输出 `ContextSegment | null`
- 上层用 `buildDefaultPhase2_1Sources` 组合，agent 内部按 priority 降序拼接

## 三、关键技术决策的原理

### 3.1 Provider 三套：为什么要 main / aux / embedding 分开？

**问题**：如果只有一个主 Provider，每次做"话题识别"、"召回意图精判"、"反思摘要"这类**廉价任务**也都走主模型，成本和延迟会爆炸。

**解法**：抽象三套 Provider 配置，各自可独立指定模型：
- `main`：用户看到的主对话（通常 qwen-plus / qwen-max）
- `aux`：辅助任务（qwen-turbo 更便宜，质量足够）
- `embedding`：向量化（专门的 embedding model）

**"复用主 Provider"开关**：用户嫌麻烦不想配三份时，勾一下就把 aux / embedding 指向 main 的 Provider 实例（共享 apiKey/baseURL，只是模型名可能不同）。

**架构上的收益**：`LLMProvider` 与 `EmbeddingProvider` 是两个独立接口。未来任何一套换家（比如 aux 换 OpenAI，embedding 换本地模型）都不用动主对话链路。

### 3.2 辅助 LLM 调用链：为什么 "粗判 → 精判 → 向量" 三段式？

**问题**：如果每条用户消息都去向量库跑一次召回，既浪费算力，又有"误召回污染主 prompt"的风险。

**现实观察**：大部分用户消息不需要召回（"这段代码什么意思"、"帮我总结下"），只有少部分需要（"上次我们聊到的那个方案"、"之前我提到过 X 吗"）。

**三段之间是"前置条件链"，不是"并行降级"**：上一步必须通过，下一步才会执行；任一步判定"不用召回"或"召回失败"，整条链立即终止，返回空结果。

#### 完整决策树（`mode='auto'`，`RelevantMemorySource` 每次用户 send 都走这条）

```
用户消息
  │
  ├─ ① 粗判 detectRecallTrigger(query)  [正则, 0 成本]
  │   │
  │   ├─ 未命中（无"上次/之前/还记得"等关键词）
  │   │    🛑 stage='keyword_miss'  终止，不调 aux、不调向量库
  │   │
  │   └─ 命中 ✓
  │       │
  │       ├─ ② 精判 callAuxIntent(query, historyHint)  [1 次 aux 调用, ~200-800ms]
  │       │   │  （没有 aux Provider 时跳过，见下方"反直觉细节"）
  │       │   │
  │       │   ├─ ANSWER: no
  │       │   │    🛑 stage='intent_no'  终止，不调向量库
  │       │   │
  │       │   ├─ aux 抛错 / 超时 / AbortError
  │       │   │    callAuxIntent 内部 catch 统一降级为 no
  │       │   │    → 和 ANSWER:no 同路径终止
  │       │   │    （设计意图：aux 不稳定时宁可不召回，也不拖累主对话）
  │       │   │
  │       │   └─ ANSWER: yes → 继续
  │       │
  │       └─ ③ 向量召回 memory.recall({semantic, types:['visit_summary'], limit:3})
  │            │  [1 次 embed + DB 查询, ~300-500ms]
  │            │
  │            ├─ 抛错   → stage='error',   hit:false
  │            ├─ 空结果 → stage='empty_result', hit:false
  │            └─ 有结果 → 邻居拼接（同 visitId 前后 2 条 episodes_msg）
  │                       → stage='success', hit:true, matches[]
```

#### `mode='explicit'`（`/recall` 命令 / `recall_memory` tool）

```
用户消息 / LLM query
  │
  ├─ 跳过粗判（绕过代码正则）
  ├─ 跳过精判（绕过 aux LLM）
  │
  └─ 直接向量召回 → error / empty_result / success
```

用户或主 LLM 已经**有意识地**表达了召回意图（显式命令、主动调 tool），不需要再用粗判/精判做"是否要召回"的二次确认。

#### 两层降级的设计语义

这里容易混的是"降级"有两重意思，在代码里**都在用**，但服务不同目标：

**降级层 A · 漏斗式性能降级（保护算力）**

| 阶段 | 命中才继续 | 未命中 | 成本 | 拦截理由 |
|---|---|---|---|---|
| ① 粗判 | 有关键词 | `keyword_miss` | 几微秒 | 大部分用户消息是闲聊 |
| ② 精判 | `ANSWER:yes` | `intent_no` | ~200-800ms | 关键词命中但语境不需要（"下次别提上次的事"） |
| ③ 向量 | 有结果 | `empty_result` | ~300-500ms | 真的没相关记忆 |

每一步尽早说"不用召回"，就能省掉后续更贵的步骤。

**降级层 B · 故障降级（保护主对话）**

任何一步抛错/超时，**绝不**让错误传播到主对话：
- `callAuxIntent` 内部 `try/catch` 把任何异常（包括 AbortError、ProviderError、网络错误）都降级成 `{intent:'no', confidence:0}`
- `memory.recall` 抛错 → `stage:'error'` 但返回空 matches，不抛出
- 主 LLM 永远拿到**空段落**，不会因召回链故障而 prompt 出错

两层降级的组合让整条链拥有**"悄无声息地失败"**的能力：用户可能永远感觉不到召回没工作，但主对话一定继续跑。

#### 反直觉细节：没有 aux Provider 时的行为

如果用户没配辅助 Provider、也没勾"复用主 Provider"开关，`deps.aux === null`。此时 `mode='auto'` 会**跳过精判**（代码见 `recall.ts` L104：`if (deps.aux) { ... }`）：

```
粗判命中 → (跳过精判) → 直接向量召回
```

- **代价**：每个命中关键词的消息都查向量库（多做些 DB 工作，但**不会**多调 LLM）
- **好处**：用户能少配一层 Provider 就用上自动召回
- **控制**：`createRelevantMemorySource(memory, aux, { enableAuxIntent: false })` 也能达到同样效果——给"我想召回但不想多花 aux 调用"的用户留的开关

#### 设计要点回顾

- **三段是前置条件链，不是并行降级**：上一步通过才进下一步
- **两层降级并存**：性能降级（早终止）+ 故障降级（不抛错）
- **精判的价值**：避免"上次的早餐"这种闲聊触发召回；aux 的小成本换来主 prompt 的纯净
- **`recall_memory` tool / `/recall` 命令走 `mode='explicit'`**：跳过 ①② 直接 ③，减少 2 次本来不必要的判断

### 3.3 反思 Job：异步沉淀记忆

**问题**：如果每条用户消息到达时就生成 visit summary + embedding + 抽 Persona 候选，主对话延迟会爆炸（每条消息多 3 个 LLM 调用）。

**解法**：**反思**——在对话**结束后**异步跑这些任务。

```
PageVisit 结束（endCurrent）→ 登记 3 条 reflection_tasks（pending）:
   1. visit_summary（生成 200 字摘要 + embedding）
   2. persona_extraction（抽 Agent 长期指令候选）
   3. persona_conflict_check（v0.2.1 占位，v0.2.4+ 实装）

ReflectionScheduler 执行时机：
   - sidebar 打开 → runPending() 补跑历史
   - SW chrome.alarms 每 60 分钟 → 广播 REFLECTION_SCAN_TICK → sidebar 调 runPending()
```

**设计要点**：
- **SW 不直接跑 Dexie**：规避 SW 与 sidebar 的 IndexedDB origin 隔离风险（详见 TROUBLESHOOTING §8）。SW 只广播信号，sidebar 在线时执行
- **失败重试 3 次**：`attemptsCount` 达到 `maxAttempts` 标 `failed`，不再重试浪费资源
- **running 标志位**：防止并发重入
- **单个任务失败不影响其它**：每个任务独立 try-catch

### 3.4 PageVisit 生命周期：对话边界的抽象

**问题**：用户一次 tab 打开到关闭之间，可能在 SPA 里跳很多页面。我们需要**一个稳定的"上下文边界"概念**来决定"这些消息算同一次对话还是两次"。

**解法**：`PageVisit` 抽象。

一次 PageVisit = 一个 visitId，包含：
- `startedAt` / `endedAt`
- `url` / `canonicalUrl` / `domain` / `title` / `articleId`

**何时新建 visit**：
- sidebar mount（首次加载）
- pushState / replaceState / popstate 导致 `canonicalUrl` 变化（SPA 硬切）
- `/new` slash 命令（用户主动）

**何时**不**新建 visit**（重要）：
- 仅 hashchange（哈希路由内切章）：canonical URL 相同
- pushState 但 canonical 仍相同

**为什么 hashchange 不切 visit？**
因为 visit 切换会触发 endCurrent → 登记反思任务 → 新 visitId。如果每次章节跳转都这样，反思队列会被炸掉。v0.2.4 我们选了折中方案：**hashchange 只清 SessionTopic，不切 visit**，这样新章节会重新识别话题但共享同一次 visit 的记忆。

### 3.5 消息持久化：v0.2.3 的"盲点修复"

这是整个 v0.2 过程里**最关键的一次修漏**（详见 TROUBLESHOOTING §11）。

**问题**：v0.2.1 铺设好了所有记忆层的骨架，但 `episodes_msg` 表在生产代码路径**零写入**。整个 `memory.remember()` 在全仓库只被 `ReflectionRunner.runVisitSummary` 调用（写 visit_summary），user / assistant 消息**从未入库**。

**连锁反应**：
- 刷新页面 = 失忆（消息没存过哪有记忆）
- 反思 Job 的 `runVisitSummary` 读 episodes 永远是空，`visit_summary` 永远生成不出来
- 向量召回没有任何素材

**根因**：整个设计文档（ROADMAP §79）写清楚了"每条消息到达立即同步写 episodes_msg"，但**实现时跨了 ui/sidebar/agent/memory 四个包的边界**，没有任何一个包觉得"写消息"是自己的事。

**修复**：
1. **`useStreamingChat` 加 `persistMessage` port**（鸭子类型，不依赖 memory）
2. **sidebar 闭包装配**：注入当前 visitId / canonicalUrl / domain / orderInVisit 作为闭包变量
3. 每条 user / assistant 消息成功产生时调用一次

**启示**（已落成编码规范）：
> 新增"需要落库的数据流"时，强制问一个问题：
> **"在生产代码里，哪一行会调用 `.remember()`？"**
> 答不上来就是这个 bug 的翻版。ROADMAP 里的"同步必做"清单应当是 PR checklist。

### 3.6 刷新预热：从"三段式 fallback"到"意图驱动"（v0.2.5 回退）

> 这一节讲一个**设计决策演化**的故事：v0.2.3 引入了一套 rehydrate 机制，v0.2.5 把它
> 撤回了。中间的推理过程比结论本身更有价值。

#### v0.2.3 的原设计：三段式 fallback

**当时的问题认知**：消息落库修复后，刷新页面 `useStreamingChat.messages=[]`，
`ChatHistorySource` 看不到历史 → 用户会感觉"AI 又失忆了"。

**当时的解法**：mount 时按优先级 fallback 拉取上下文注入到 LLM：

```
档 1 · WorkingMemory：已由 WorkingMemorySource 自动注入 system 段（保留）
档 2 · 跨 visit 近期消息：按 canonicalUrl 拉最近 10 条，塞进 initialHistoryForLLM
档 3 · 向量召回：由 RelevantMemorySource 在用户提问时触发
```

**UX 关键设计**：预热历史只喂 LLM、不进 UI `messages[]`——对应"不把笔记摊开在脸上"的哲学。

#### v0.2.5 发现的问题：违反"意图驱动"原则

真机跑一段后发现 **档 2 的自动预热是错的**。用户一句话点破：

> "真正需要预热的应该只有 WorkingMemory。只有当用户问'我们今天做了什么'、
> '我们这几天学了哪些'时，才去考虑跨 visit 搜索（而且搜的不只是会话消息，
> 还应该有对应的事件/摘要信息，跨 visit 按时间轴查找）。
> 向量召回则是当用户提到'我们上次 XXX'时才触发。"

这个观察命中了档 2 的三个真实问题：

1. **默认行为就是错的**：大多数场景用户刷新是为了"继续做当前事"，
   WorkingMemory 已经足够指示"我们在做什么"。额外塞历史消息等于**干扰**——
   浪费 token + 可能让 LLM 把旧对话当作当前对话一部分
2. **范围选错了维度**：按 `canonicalUrl` 查是"同 URL 的过去访问"；
   但"今天学了什么"这种元查询需要的是**时间轴 + 跨 URL**。
   维度的选择**应该由 query 决定**，不是 mount 时强制选一个
3. **数据不全**：时间维查询需要的是"事件级"信息（visit_summary 摘要），
   不是原始消息。档 2 拉的只是 `episodes_msg`

本质上，v0.2.3 档 2 用**"启动时无脑预热"**冒充"意图驱动的召回"——
刚好让用户错觉"AI 记得我"，但代价是每次对话都在拖一条无关的历史。

#### v0.2.5 的修正：恢复纯粹的三档意图驱动

```
默认（大多数场景）：
    档 1 · WorkingMemory  → WorkingMemorySource 自动注入 system
                           （activeGoal + TODO，"我们在做什么"）

用户意图时：
    档 2 · 时间维元查询    → Chronological Index（ROADMAP 登记中）
         ("今天/本周看了什么")   查 episodes_visit_summary + 跨 visit 按 timestamp

    档 3 · 语义历史指代    → RelevantMemorySource 向量召回
         ("上次/之前的 XX")     查 episodes_visit_summary + 邻居 episodes_msg
```

**实现**：删除 `sidebar/index.tsx` 里的 rehydrate useEffect；`initialHistoryForLLM` port
在 `useStreamingChat` / `ChatPanel` 保留——因为 Chronological Index 能力落地时，
时间维查询的结果仍会通过这个 port 注入（只是触发条件变了）。

**不变的部分（关键）**：`persistMessage` **完全保留**。落库是**数据沉淀**，预热是**数据注入**——
两者独立：

| 机制 | 作用 | v0.2.5 状态 |
|---|---|---|
| `persistMessage`（落库） | 每条消息写入 `episodes_msg`，供未来检索的**基础数据** | ✅ 保留 |
| rehydrate 注入（档 2） | mount 时把历史塞进 LLM 的 history | ❌ 撤回 |

#### 接受的代价

**刷新后"承接式失忆"**：用户刷新前聊到一半，刷新后说"然后呢"这种承接输入——
没有意图线索能触发 recall，LLM 不会自动拿到上轮对话。用户要补一句上下文。

这是**有意的取舍**：
- 大多数场景（~95%）不受影响，上下文反而更纯净
- 极小比例的承接式失忆用户一句话能修复
- 如果真机发现此场景高频，再加"30 分钟内同 visit 的 episodes_msg 轻量兜底"的窄条件预热（已登记 ROADMAP v0.2.5+）

#### 从这次演化里得到的教训

**"便利性" vs "正确性"的经典权衡**：
- v0.2.3 档 2 是一个**便利性驱动**的设计：启动就把所有可能有用的东西预热进去
- v0.2.5 撤回它，选择**正确性驱动**：只在用户意图明确时才动手

类似的陷阱在 AI 产品里特别多——"让模型记得更多"听起来总是对的，
但实际上**"让模型在正确时刻记得正确的东西"**远比"塞更多进去"重要。
召回系统的首要指标不是召回率，是**精准度与相关性**（不相关的召回会主动污染推理）。

### 3.7 Persona 语义转向：v0.2.2 的一次设计反转

**v0.2.1 的原设计**：Persona = "关于用户的稳定事实/偏好"（例："用户偏好 TypeScript"、"用户是前端"）。

**真机暴露的问题**：千问自发调用 `remember_persona` 时写的是"**我是小瑾，用户专属的文档助手...**"——这是**自我设定**，不是"关于用户"。模型的直觉明显是"记下我该怎么做"，而不是"记下用户是什么样"。

**v0.2.2 的解法**：不改 schema，只做**语义重定向**——Persona = "**Agent 应当长期遵守的指令 / 行为规则**"。

具体改法：
- 用户说"我是前端" → 不再写"用户是前端工程师"，而是转译为"**回答时默认使用前端语境举例，不必解释基础 Web 概念**"
- 用户说"叫我小瑾" → 写"**称呼用户为小瑾**"
- 用户说"你是我的文档助手" → 写"**你的身份是小瑾的文档助手**"

所有内容都是**写给 Agent 自己看的陈述/祈使句**。

**为什么这么改**：
1. 模型的直觉对齐：不用再做两层认知翻译，直接写"我该怎么做"
2. PersonaSource 注入话术对齐：system 段标题从"关于用户的长期记忆"改为"**你的长期指令（用户已确认的行为规则）**"
3. 审核体验更好：用户看"称呼你为小瑾"比看"用户叫小瑾"更容易判断接不接受

**v0.2.4 的二次复审**：用户发现两种形态（用户画像 vs Agent 指令）的用法各有价值——"我是前端"有时候也想被记成事实而不是规则。所以 v0.2.4+ 的 ROADMAP 重新登记了"Persona 双主体 agent/user 区分"方案，但这次会**加 `subject` 字段**在同一张表维护两类，而不是再做一次单一化反转。

### 3.8 上下文分层机制的真正落地（v0.2.4）

用户在 v0.2.3 末期提了一个犀利观察：
> "上下文是 session（所见即所得）/ WorkingMemory / TopicMemory / PersonalMemory + 按需召回。
> **核心是：上下文不是跟会话强绑定的**——聊着聊着话题变了就更新 topic，下次用新的 topic。"

**v0.2.4 查下来发现的真相**：架构上四层都有，但两个关键"可变性"没落地：
1. **SessionTopic 识别函数写好了但从未被生产代码触发**（只有 `/topic` 命令会调）——详见 TROUBLESHOOTING §12
2. **hashchange 既不监听，canonicalizeUrl 又会剥 hash**——切文章 Agent 不知道——详见 TROUBLESHOOTING §13

**v0.2.4 修复**：
- `useStreamingChat.onRoundFinished` port → sidebar 用 `shouldIdentify(count)` 判定 → 每 4 轮自动调 `identifySessionTopic`
- hashchange 监听 → 清当前 visit 的 topic（不切 visit）→ 下一轮 `shouldIdentify(1)=true` 立即重新识别

**配合 `UIMessage.visitId` 标签 + `groupMessagesByVisit`**（问题 4）：SPA 内切文章后，旧 visit 的消息会被自动降级成 system 段`# 之前在《上篇文章》中的对话` + 明确降权提示，不再污染新文章上下文。

### 3.9 Slash 命令：让用户有"手动覆盖"的通道

虽然我们追求"Agent 像真正的助手一样自动工作"，但用户仍然需要偶尔**手动干预**。v0.2.1 新增了三个命令：

| 命令 | 作用 | 哲学 |
|---|---|---|
| `/new` | 清 UI + 新 visitId，**不清** WorkingMemory / Persona / Episodic | 开新章节但不失忆 |
| `/recall <query>` | 显式召回；绕过 auto 模式的粗判，直接语义向量 | 主 LLM 不主动找时用户自己找 |
| `/topic [<text>]` | 无参：强制触发一次主题识别；有参：手动设置 topic | 话题自动识别漂移时的救济 |

命令的 UX 设计原则：
- 命令菜单即时响应，不等 LLM
- `/recall` 命中后，结果以**非流式 assistant 消息**形式 append 到聊天流（通过 `appendAssistantNote`），下一轮 LLM 能看到
- 所有命令能力通过**可选 port**注入 `SlashCommandContext`，降级友好（port 未注入时命令提示"不支持"而不是崩）

### 3.10 UI 层的"透明"设计

整个 v0.2 UI 改动都围绕一条准则：**让 Agent 的智能显现为结果，而不是显现为界面**。

- 反思结果产出 Persona 候选 → 折叠条浮现，一键采纳/忽略（不弹大窗）
- WorkingMemory 有 activeGoal → 顶部小卡片（可折叠），不是全屏 Todo 面板
- 刷新时的预热 history → **不进 UI**，只喂 LLM（v0.2.3 核心设计）
- tool-call 默认不展示（预留 v0.2.4+ 加徽章）

**反例修复**（v0.2.4 问题 5、6）：
- WorkingMemoryCard 在某边界展开只有动画 → 必须让用户看到真实内容
- PageContextCard 永远只显"201字摘要" → 必须让用户能看到真实摘要

"让用户看不到状态"和"让用户能信任系统"是两件事。前者是 UX 简洁，后者要求**关键信息随时可验证**。v0.2.4 的两处 bug 正好踩在这条边界上。

## 四、踩过的坑与形成的编码规范

### 4.1 "架构里有 ≠ 真的在跑"

**两次出现**：
- v0.2.1 铺了记忆骨架但 `episodes_msg` 零写入（TROUBLESHOOTING §11）
- v0.2.1 铺了 `identifySessionTopic` 但生产代码从未自动触发（TROUBLESHOOTING §12）

**规范**：新增任何"需要被触发的函数"时，PR checklist 必包含：
> **"在生产代码里，哪一行会调这个函数？"**
> 答不上来 = 不能合并。

### 4.2 Tool description 的主体必须显式

v0.2.1 `remember_persona` description 模糊，被模型误用为写"我是小瑾..."（TROUBLESHOOTING §10）。

**规范**：tool 的 description **必须显式写明**：
- "谁"调（主 LLM？反思 Job？）
- "关于谁"（用户？Agent 自己？当前页面？）
- "什么时候调"（主动触发时机）
- "什么时候不调"（反例）

v0.2.3 据此把所有 WorkingMemory / 记忆类 tool 的 description 全面重写，每个都有"主动触发时机 / 不要调用的情况 / 格式示例"三段。

### 4.3 `exactOptionalPropertyTypes` 的代价

v0.2.1 真机 typecheck 时爆了一大堆错（`displayReason: undefined` 不合法）。

**规范**：本项目 `tsconfig` 开了 strict，optional 字段 `x?:` 不接受显式 `undefined`。

约定写法：
```ts
// ❌ 会 typecheck 失败
const rec = { a: 1, b: maybeB ?? undefined };

// ✅ 条件展开
const rec = { a: 1, ...(maybeB !== undefined ? { b: maybeB } : {}) };
```

### 4.4 分层不允许反向依赖

`@doc-assistant/ui` 不能依赖 `@doc-assistant/agent`；`@doc-assistant/tools` 不能依赖 `@doc-assistant/agent`；`@doc-assistant/memory` 不能依赖 `@doc-assistant/agent`。

**约束方式**：
- ESLint 规则约束（`@doc-assistant/*/no-reverse-deps`）
- 跨层需要类型时用**鸭子类型**在本包定义（例如 tools 里的 `PageVisitLike`、ui 里的 `PersonaView` / `WorkingMemoryView`）
- port 模式注入：ui 不知道 MemoryStore 的存在，只接受一个 `persistMessage: ({role, content}) => Promise<void>` port

### 4.5 Provider 的 "usage" 字段白送

千问的 stream finish part 带了 `usage.promptTokens / completionTokens / reasoningTokens`，已在 `normalizer.ts` 归一化到 `ChatChunk.usage`。Token 看板（v0.2.3+ 待办）可以直接消费，不用二次请求。

## 五、设计沉淀：系统里的"可变上下文"模型

v0.2.5 之后，我们的上下文注入机制是这样工作的：

```
每轮 send():

    initialHistoryForLLM  (v0.2.5 起默认为空；
                           port 保留供未来 Chronological Index 注入时间维召回结果)
           ↓
    groupMessagesByVisit(messages, currentVisitId)
        ├─ 当前 visit 消息 → 原文透传
        └─ 其它 visit 消息 → 按 visit 分组，每组前置:
             "# 之前在《上篇文章》中的对话（N 条）
              以下是用户在切换到当前文章之前的对话，仅作为背景参考..."
           ↓
    [真正要发的 ChatMessage[]]
           ↓
    agent.run() 内部 sources 按 priority 拼装:
        system(100) = DEFAULT_CHAT_SETTINGS.systemPrompt
        system(80)  = page-context (title + summary + "需要原文请调 tool")
        system(70)  = reference-tag (用户划词)
        system(60)  = persona (长期指令 bullets)
        system(55)  = session-topic (当前焦点)
        system(50)  = working-memory (activeGoal + TODO)  ← 默认场景"我们在做什么"
        system(40)  = relevant-memory (auto 召回命中时)    ← "上次/之前" 触发
        user/assistant = chat-history (ctx.history)
           ↓
    LLM
```

**"可变性"体现在**：
- Persona 可被新 remember_persona / 反思 Job 更新 → 下次注入就变
- SessionTopic 会每 4 轮自动识别；hashchange 清空后下轮重识别
- WorkingMemory 会被 LLM 通过 tool 动态维护
- 召回的 visit_summary 随新访问动态扩充

**"意图驱动"体现在（v0.2.5）**：
- 默认刷新：只靠 WorkingMemory（activeGoal + TODO）告诉 LLM"我们在做什么"
- 用户说"上次 / 之前"：`RelevantMemorySource` 自动走向量召回
- 用户说"今天 / 本周学了什么"（规划中）：走 Chronological Index，按时间轴跨 visit 拉 visit_summary
- **不主动预热不相关的历史** —— 避免污染纯净上下文

**用户核心洞察（v0.2.3 末期）**：
> 上下文不是跟会话强绑定的。聊着聊着话题变了就更新 topic；下次用新 topic。

这个愿景在 v0.2.4 落到了代码里。

## 九、给未来自己的话

1. **接下来最优先做的**：根据真机使用感受决定 v0.2.5+ 的优先级。当前队列里 **Token 看板**（观测性）和 **跨 visit 时间维检索 · Chronological Index**（功能缺口，且和 v0.2.5 撤回的 rehydrate 是同一件事的正确做法）是用户直接反馈的，优先做它们。
2. **不要再做 Persona 语义反转**：v0.2.2 已经反过一次，v0.2.5+ 的 `subject` 字段是"**叠加**"而不是"**替换**"——同一张表两类视角并存，不要再把 v0.2.2 的转译规则推翻。
3. **添加新记忆类型前**：先问"哪条 ContextSource 注入它？priority 多少？什么情况下返回 null？" 写不出来就说明还没想清楚。
4. **新 tool 前**：强制先写"主动触发时机 / 不要调用的情况 / 格式示例"三段 description（哪怕只是写给 `description:` 字符串里面），模型会照做。
5. **UI 层改动前**：再读一遍 §3.10 的"透明设计"哲学。加 UI 总是容易的，但每加一块都在稀释"真正的助手"这个感觉。
6. **记忆相关改动前（v0.2.5 沉淀）**：严格区分"**数据沉淀**"（`persistMessage` 写 `episodes_msg`）与"**数据注入**"（往 LLM 的 messages 里塞东西）。前者是基础，缺了什么都做不了；后者必须**意图驱动**——用户没表达意图就不要自作主张"为了让 AI 有记忆"把一堆东西塞进去。"让模型记得更多" ≠ "让模型在正确时刻记得正确的东西"，后者才是目标。

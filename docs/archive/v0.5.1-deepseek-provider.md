## 0. 背景与目标

### 现状

- `packages/provider/` 自 v0.2 定稿起只实现了 `QwenProvider` + `QwenEmbeddingProvider` 一家；三套 Provider 配置（`main` / `auxiliary` / `embedding`）**schema 层面已是通用的** `baseUrl + model + apiKey`，但**具体适配只有阿里云一家**。
- 用户群体里对 DeepSeek 的呼声明显高于其它候选（OpenAI / Anthropic / Ollama）：V3 / R1 系列在中文开发者场景里是当前成本 / 效果比最优的选择之一；DeepSeek 的 API 基本兼容 OpenAI 协议，接入成本可控。
- ROADMAP · §6 虽把"Provider 层抽象"列为延后项，但 DeepSeek 属于**高 ROI、低风险**的单点落地，适合在 v0.5.x 小版本补丁位先走通"第二家 Provider"的全流程，为后续 OpenAI / Anthropic / Ollama 等的陆续接入打样。

### 为什么做（用户价值）

1. **辅 LLM 降本**：SessionTopic 识别、反思 Job、召回 intent 精判这些高频低难度任务，用便宜模型跑比复用主模型经济得多——DeepSeek 的价格带非常合适。
2. **推理场景增强**：R1 系列的深度推理能力能让"反思 Job / Persona 抽取 / 冲突检测"的质量上一个台阶。
3. **用户选择权**：填 API Key 即可切换 Provider，不改代码、不重装扩展。
4. **为 B-002 / B-003 / B-004 铺路**：自定义智能体、subAgent、Skills 一旦落地，必然会有"跑一堆辅助小任务"的需求，DeepSeek 的价格带是最合适的"辅助算力燃料"。

### 目标（一句话）

**让用户通过配置页下拉选择 "DeepSeek"、填入 API Key 和模型 ID，即可把 `main` / `auxiliary` / `embedding` 中的任意一路或全部切换到 DeepSeek，所有下游消费方（sidebar / ReflectionRunner / RelevantMemorySource 等）无感继续工作。**

### 本期边界（一句话）

本期**只做 DeepSeek 一家**的接入；不顺手做 OpenAI / Anthropic / Ollama；不重构 Provider 层抽象（保持现有"类 OpenAI 兼容协议 + 千问特例"的现状，DeepSeek 自然落在前者即可）。

## 1. 用户故事

### US-1 · 第一次使用便宜辅助模型（主用户故事）

> 作为一个日常使用 Doc Assistant 的用户，我已经在**主对话**里用千问用得很开心；但我发现扩展每次后台跑反思 Job 和主题识别也在吃千问 token，成本偏高。
> 我希望**只切换"辅助 LLM"这一路**到 DeepSeek（保留主对话用千问），用 `deepseek-chat` 或更便宜的型号，这样反思 Job 这类批量小任务成本能降一个数量级，且不影响主对话质量。
>
> 完成标志：配置页"辅助"那组 Provider 下拉里能选到 "DeepSeek"，填 key + baseURL（有默认值）+ 模型 id 后保存，下一次 PageVisit 结束的反思 Job 就能跑在 DeepSeek 上，真实 token 消耗在用量看板（若已上线）或 DeepSeek 控制台里可见。

### US-2 · 主对话换 R1 做深度推理

> 作为一个**偏技术/研究场景**的用户，我在读 arXiv 论文时希望助手的推理深度更高，尤其是"这个方法和另一篇的差异"这种对比分析。
> 我希望把**主对话**切到 DeepSeek 的 `deepseek-reasoner`（R1），在关键问题上拿到带思维链的回答；同时保留辅助 LLM 用便宜型号。
>
> 完成标志：配置页"主"那组 Provider 下拉里能选到 "DeepSeek"、填模型 id 为 `deepseek-reasoner` 后保存，sidebar 里发起对话，能看到流式正常、工具调用正常、usage 正常统计；`ReflectionRunner` 仍使用我在"辅助"组配的 Provider（例如仍是千问 Turbo），互不干扰。

### US-3 · 零停机切换 Provider

> 作为一个**已有历史记忆**的老用户，我的 `episodes_visit_summary` 都是用千问 v3 embedding 建的索引。我希望切换**主 / 辅**两路到 DeepSeek 时，**不破坏**已有历史记忆的可召回性；如果 DeepSeek 暂无官方 embedding 服务（或我不想换），我希望 embedding 那一路能**保留为千问**不被迫换。
>
> 完成标志：配置页三路 Provider 的 "DeepSeek" 选项**可分别独立启用**；任意一路切换到 DeepSeek 保存后，sidebar 重载即生效，已有 `episodes_visit_summary` 能继续被 `recall_memory` 召回（因为 embedding 路径未改变）；若用户选择不支持 embedding 的 Provider，配置页给出**明确的 UI 提示**。

## 2. 功能范围

### In-Scope（必须完成）

1. **Provider 下拉新增 "DeepSeek" 选项** — 出现在 `main` / `auxiliary` / `embedding` 三套 Provider 配置的下拉里，与 "Qwen" 并列。
2. **三条链路分别可用**：
   - `main`：主对话（流式、tool calling、usage 统计全部正常归一化到现有 `ChatChunk` 契约）
   - `auxiliary`：反思 Job、SessionTopic 识别、Intent 精判（非流式调用，结构化输出）
   - `embedding`：向量化 `visit_summary` 与 query。若 DeepSeek 当期无官方 embedding 服务，**此路可标为"不适用"**并在 UI 上明确提示"DeepSeek 暂不提供官方 embedding，请选择其它 Provider（如 Qwen）"
3. **默认值**：用户选择 DeepSeek 后，`baseUrl` 自动填入 DeepSeek 官方默认值（可编辑覆盖，兼容自托管），`model` 输入框给占位提示（如 `deepseek-chat` / `deepseek-reasoner`）。
4. **兼容现有契约**：不修改 `MemoryStore.remember/recall`、`LLMProvider`、`EmbeddingProvider`、`ChatChunk`、`ContextSource` 等任何跨层契约；DeepSeek 适配**内含**在 `packages/provider/` 内完成。
5. **错误归一化**：DeepSeek API 的失败路径（auth / rate-limit / 模型不存在 / 网络）统一映射到现有错误类型，sidebar 报错 UI 文案一致，不引入 DeepSeek 特有的错误语义。
6. **不破坏 Qwen**：现有 Qwen 用户无需做任何操作，升级后配置原样工作；既有单测全绿。
7. **README "支持的模型" 章节** 同步更新，列出 DeepSeek 为"已支持"。
8. **ROADMAP · §6** "Provider 层抽象以支持 OpenAI / Anthropic / Ollama 等" 的延后项**同步勾选**（标注"DeepSeek 已于 v0.5.1 打样接入，其余保持延后"），不整体勾掉该条。

### Out-of-Scope（本期明确不做）

1. **Provider 层大重构**：不做 `BaseOpenAICompatibleProvider` 抽象类、不做 Provider 插件化注册机制——那是未来单独立项（见"风险与开放问题 R-3"）。
2. **OpenAI / Anthropic / Ollama 的一并接入**：即使 DeepSeek 和 OpenAI 协议兼容度高、技术上"顺手能做"，本期也**不做**；避免范围蔓延。
3. **DeepSeek 特有能力的深度绑定**：
   - R1 的"思维链"字段：**已于 2026-05-06 由 PM 拍板采纳 C 方案** —— 新增 `ChatChunk.reasoning` chunk 类型，UI 侧以折叠气泡形式展示思维链（详见 §6 R-2 与本章最后"范围调整说明"小节）。这意味着本期需要在 `ChatChunk` 契约上新增一种 chunk 类型，涉及 Provider 层到 UI 层的跨层契约扩展。
   - DeepSeek 的 `Prefix` / `FIM` / Beta 接口 —— 本期**不接**
4. **DeepSeek 官方 embedding（如官方后续发布）**：发布后再评估；本期只写"不适用"分支。
5. **Provider 级的 token 用量看板**：用量看板本身是 ROADMAP `v0.2.5+` 的独立项，本期**不做**，但本期的 Provider 实现要**预留** usage 字段归一化，以便看板项能无缝消费（见详细验收 §5.1-c）。
6. **配置迁移脚本**：v0.1 → v0.2 的迁移已做过，本期新增 DeepSeek **不改动**已有 Qwen 用户的配置结构（只在原 schema 的 `kind` 枚举中新增一个值），不需要迁移。

### 范围调整说明（2026-05-06 · R1 `reasoning_content` 采纳 C 方案后新增）

- **背景**：R-2 原默认项为 A（丢弃），明确标注"C 方案（新增 `ChatChunk.reasoning` 类型）**超出本期范围**（要动跨层契约）"。2026-05-06 PM 拍板改采 C 方案后，本期范围事实上**扩大**为"DeepSeek 基础接入 + `ChatChunk.reasoning` 契约扩展 + UI 折叠展示思维链"。
- **影响面（一览）**：
  - `packages/provider/`：DeepSeek 适配器在流式过程中把 `reasoning_content` 以 `reasoning` chunk 发出（Qwen 侧若无对应字段，保持不发 `reasoning` chunk 即可，向后兼容）
  - `ChatChunk` 契约：枚举新增 `'reasoning'` 成员，payload 至少含 `text: string`（可增量追加）
  - `packages/agent/`：Loop 层识别 `reasoning` chunk 但**不混入** `episodes_msg` 正文（不入库 / 不进 ChatHistory 注入）
  - `packages/ui/`：`MessageBubble` 侧新增"思考过程"折叠块（默认折叠、用户可展开），视觉风格可复用既有 `ThinkingBlock`
  - 单测：Provider / Agent / UI 三层分别补一条最小用例
- **两条路径留给用户后续确认**：
  - **(a) · 保持 v0.5.1 范围，把 `ChatChunk.reasoning` 作为前置改动一并承接**（R1 是 DeepSeek 的招牌特性，少了它本期价值削弱；契约扩展本身是"只加不改、向后兼容"，风险可控）。
  - **(b) · 拆出一个 v0.5.2（或 pre-v0.5.1）做契约扩展，v0.5.1 只做 DeepSeek 基础接入（R1 思维链暂丢弃），等契约就绪后再开启 `reasoning` 通路**（好处：v0.5.1 范围最小、可更快发布；代价：DeepSeek R1 用户首次上线体验 = 看不到思维链，需再等一个小版本）。
- **PM 推荐：(a)**。理由：R1 是 DeepSeek 的招牌能力，"只接 DeepSeek 却看不到 R1 思维链"会让本期用户价值被显著削弱；契约扩展属于"新增枚举 + 新增 payload"的向后兼容扩展，既不破坏 Qwen，也不阻塞未来 Provider，工时估计在原 6–10h 之上追加 2–4h 可承接。
- **待确认**：上述 (a) / (b) 的最终选择留给用户后续拍板；在用户确认前，本文档 §2 In-Scope / Out-of-Scope 暂按 (a) 路径书写（即"本期承接契约扩展"）。若用户选 (b)，需要回写 §2、§3 相关 AC、§6 R-2 与本小节。

## 3. 详细验收标准

> 验收视角：下表每一条都是**可由 developer 自检、也可由 PM 在测试构建上亲手验证**的具体行为。

### 3.1 配置页 UI

- [ ] **AC-UI-1**：配置页 `main` / `auxiliary` / `embedding` 三组 Provider 表单的 "Provider" 下拉选项里，**"DeepSeek" 与 "Qwen" 并列**；选项文案统一（如均为中文"DeepSeek / 通义千问"或均为英文，保持现有风格一致）。
- [ ] **AC-UI-2**：用户在任一组下拉选中 "DeepSeek" 后：
  - `baseUrl` 输入框**自动填入 DeepSeek 官方默认值**（可编辑覆盖）
  - `model` 输入框**不自动填**具体型号（避免默认值造成用户误配），但 placeholder 给出示例：`如 deepseek-chat 或 deepseek-reasoner`
  - `apiKey` 输入框**保持空**，等用户粘贴
- [ ] **AC-UI-3**：切换 Provider 下拉时，已填的 `apiKey` **不被自动清空**（避免用户误切回原 Provider 时要重输）；但如果 `baseUrl` 当前是旧 Provider 的默认值，则**提示替换**为新 Provider 的默认值（一个"使用 DeepSeek 默认 baseUrl"的小按钮或弹窗二选一即可，不强制）。
- [ ] **AC-UI-4**：`embedding` 组选中 "DeepSeek" 时，如果 DeepSeek 当期不提供官方 embedding 服务，UI 上给出**一条清晰的提示**（如：`⚠️ DeepSeek 暂不提供官方 embedding 服务，建议选择 Qwen。若你使用兼容协议的自托管服务，请确认 model 参数正确。`）；用户依然**可以保存**该配置（为自托管场景留余地），但 sidebar 加载时遇到 embedding 调用失败会降级（走现有 embedding-null 链路：只影响向量召回，不影响主对话）。
- [ ] **AC-UI-5**：配置页的"测试连接"按钮（如果当前已有该按钮）对 DeepSeek 同样生效——点击后能看到成功 / 失败反馈，失败时带可读错误信息。**如果当前没有此按钮**，本期不新增，属 Out-of-Scope。

### 3.2 主对话链路（`main`）

- [ ] **AC-MAIN-1**：用户把 `main` 切到 DeepSeek（`deepseek-chat`）后，在 sidebar 发起普通对话（无工具调用），流式输出**逐字显示**无卡顿，结束后 `assistant` 消息完整落库（`episodes_msg`）。
- [ ] **AC-MAIN-2**：触发至少一个 tool-calling 链路（如 `get_working_memory` / `recall_memory`），LLM 能正确产出 tool_call、tool 执行结果回灌后 LLM 继续产出最终回答，**全流程与 Qwen 行为一致**。
- [ ] **AC-MAIN-3**：usage 字段（`promptTokens` / `completionTokens` / 若 API 返回 `reasoningTokens`）能从 DeepSeek 响应里**正确提取并归一化**到 `ChatChunk.usage`；若字段缺失则填 0，不抛异常。
- [ ] **AC-MAIN-4**：切到 `deepseek-reasoner`（R1）时，主对话同样可用。**响应中的 `reasoning_content`（思维链）必须通过新增的 `ChatChunk.reasoning` chunk 类型流出**，不得被合并进 `text` chunk、不得被丢弃；Loop 层不得把 `reasoning` chunk 拼进 `episodes_msg` 正文（保证入库消息洁净、后续注入 ChatHistory 也不污染）。
- [ ] **AC-MAIN-4b · UI 折叠展示**：UI 侧（`MessageBubble` / 思考过程折叠块）默认把 R1 思维链**折叠收起**，用户点击可展开查看完整推理过程；折叠态与 assistant 正文在视觉上可区分（如浅灰背景 + "思考过程"标签），不影响主回答的阅读节奏。
- [ ] **AC-MAIN-5**：对 DeepSeek 常见失败（401 auth / 429 rate-limit / 404 model / 5xx / 网络超时）sidebar UI 的报错形态与千问一致——**同一个错误展示组件、同一套降级文案**，不引入 DeepSeek 特有的新组件。

### 3.3 辅助链路（`auxiliary`）

- [ ] **AC-AUX-1**：`ReflectionRunner` 中的 `visit_summary` / `persona_extraction` 任务用 DeepSeek 跑，**产出结构化 JSON 能被既有 parser 正常解析**；异常情况（JSON 不合法 / 字段缺失）走既有重试 / 降级链路，不产生新的失败类型。
- [ ] **AC-AUX-2**：`identifySessionTopic` / `callAuxIntent` 用 DeepSeek 跑，**返回延迟与 Qwen Turbo 同量级**（PM 视角不设死指标，但 developer 需在 CHANGELOG 注明真机抽样延迟），且结果结构正常。
- [ ] **AC-AUX-3**：`auxiliary` 与 `main` 可独立配置不同 Provider——例如 `main=Qwen, auxiliary=DeepSeek`，或 `main=DeepSeek-R1, auxiliary=DeepSeek-Chat`，均能正常工作；两路的调用链路互不污染（一个错误不连带另一个失败）。

### 3.4 Embedding 链路（`embedding`）

- [ ] **AC-EMB-1**：若 DeepSeek 当期**无官方 embedding**，`embedding` 下拉里的 DeepSeek 选项仍然存在（方便自托管场景），但 UI 提示明确（见 AC-UI-4）；用户选中并保存后，sidebar 加载时尝试 embedding 调用失败则**降级**（`RelevantMemorySource` 的向量召回路径静默失败，不报致命错误，不影响主对话）。
- [ ] **AC-EMB-2**：**推荐配置**在 README 里明确：`embedding` 路保留为 Qwen（官方 v2/v3），`main` / `auxiliary` 按需选 DeepSeek；文档给出"混搭配置示例"。
- [ ] **AC-EMB-3**：已有用户的历史 `visit_summary` 向量（用 Qwen embedding 建的索引）**不因 `main` / `auxiliary` 切换到 DeepSeek 而失效**——因为 embedding 路径不变，老索引继续可用。

## 5. 与现有 `packages/provider/` 体系的衔接点（接口约定级别）

> 本节**仅定义接口/契约级别的衔接要求**，不做技术选型。developer 在实装时可自由决定是否复用现有"类 OpenAI 兼容"适配器、是否新建 `DeepSeekProvider` 独立类、是否引入新依赖等，只要**对外契约**满足下列要求即可。

### 5.1 对现有契约的遵守

- **a. `LLMProvider` 接口保持不变**：DeepSeek 的 chat / stream / tool calling 能力必须通过现有 `LLMProvider` 接口暴露；不新增新方法、不修改签名。
- **b. `EmbeddingProvider` 接口保持不变**：`embedding` 路的 DeepSeek 实现（若后续支持）必须通过现有 `EmbeddingProvider` 接口暴露。
- **c. `ChatChunk` 归一化约定**：
  - 流式分块类型在现有枚举（`text` / `tool-call` / `tool-result` / `finish` / `error`）基础上**新增 `reasoning` 类型**（2026-05-06 决策）——payload 至少含 `text: string`，可增量追加；Qwen 侧无对应字段时**不发** `reasoning` chunk，保持向后兼容。
  - `usage` 字段在 `finish` chunk 里以现有键名归一化（`promptTokens` / `completionTokens` / 可选 `reasoningTokens`）
  - R1 等推理模型的 `reasoning_content` **必须通过新增的 `reasoning` chunk 流出**，不得合并进 `text` 正文、不得丢弃；UI 默认折叠展示、用户可展开；Loop 层不得将 `reasoning` chunk 内容写入 `episodes_msg` 正文或注入 ChatHistory。
- **d. 错误类型归一化**：DeepSeek 失败映射到现有错误类型，不引入 DeepSeek 特有 error class。

### 5.2 对配置 schema 的影响（收敛为"新增 kind 枚举值"）

- **现有 `ProviderConfig` 的 `kind` 字段**（或类似判别字段）在枚举中**新增 `'deepseek'`**（具体命名由 developer 与 PM 在 PR 阶段二选一：`deepseek` / `deepseek-chat`，推荐 `deepseek`）。
- 其它字段（`baseUrl` / `model` / `apiKey`）**不变**，延续"通用 OpenAI 兼容三元组"约定。
- 不新增 Provider 专属嵌套字段（即不做 `providerSpecific: { deepseek: {...} }`）；若后续真需要，另立需求。

### 5.3 装配层（bootstrap）的责任

- `apps/extension/src/sidebar/bootstrap.ts`（以及 offscreen / options 三处装配入口）需要根据 `kind='deepseek'` **选择并实例化** DeepSeek 适配器；装配失败则走现有"Provider 未就绪"降级路径（与 Qwen 缺失 key 场景对齐），不打断其余模块加载。
- 装配层**不感知** DeepSeek 的具体 SDK / HTTP 细节；这些细节完全封装在 `packages/provider/` 内。

### 5.4 观测性预留（不在本期实装，但要求契约兼容）

- `usage` 归一化要保证未来"Token 用量看板"（ROADMAP · v0.2.5+）能按 `provider-kind × model-id × 日期` 聚合——这要求 `kind='deepseek'` 的 usage 上报**和千问使用相同字段结构**，且 Provider 实例带"角色标签"（`main` / `auxiliary` / `embedding`，沿用现有 bootstrap 打标机制）。

## 6. 风险与开放问题

### R-1 · DeepSeek 官方 embedding 服务的不确定性（开放）

- **现状**：截至定稿日（2026-05-06），DeepSeek 官方未提供稳定的 embedding API。
- **本期策略**：`embedding` 路保留 DeepSeek 下拉选项（支持自托管兼容场景）+ UI 明确提示，但**推荐用户选 Qwen**。
- **后续**：若 DeepSeek 发布官方 embedding，评估是否需要独立补丁版本（v0.5.2 或并入下一大版本）。
- **需 PM 拍板**：**无需立即拍板**；若 developer 在实装期间 DeepSeek 官宣 embedding，临场决定是否顺手支持（小改动）或单独立项（大改动）。

### R-2 · R1 `reasoning_content` 的展示策略（已决策：C 方案 · 2026-05-06）

- **问题**：DeepSeek R1 的响应里可能带独立的思维链字段（`reasoning_content`）。当前 `ChatChunk` 契约里**没有**专门承载这种字段的类型。
- **选项**：
  - **A · 丢弃**：只取最终 content，思维链不展示给用户。优点：零契约变更；缺点：用户用 R1 可能就是为看思维链。
  - **B · 合并进正文**：把 `reasoning_content` 作为 `text` chunk 拼接在最终答复前，用某种视觉分隔。优点：用户能看到；缺点：正文被污染，`episodes_msg` 入库也会连带污染。
  - **C · 新增 `ChatChunk` 类型 `reasoning`**：架构最干净；代价：**动跨层契约**（Provider → Agent → UI 三层）。
- **最终决策（2026-05-06）**：**已决策：C 方案**。理由：A 会显著削弱 DeepSeek R1 的核心价值；B 会污染 `episodes_msg` 与 ChatHistory 注入链路、破坏"入库消息保持洁净"的底线；C 虽然动契约但属于"新增枚举 + 新增 payload"的向后兼容扩展，不破坏 Qwen、不阻塞未来 Provider。具体承接细节见 §2 末尾"范围调整说明（2026-05-06）"小节与 AC-MAIN-4 / AC-MAIN-4b / §5.1-c。
- **本期范围联动**：本期范围**已扩大**承接 `ChatChunk.reasoning` 契约扩展；是否拆到 v0.5.2 独立承接（方案 b）留给用户后续确认，PM 推荐保持 v0.5.1 内一并承接（方案 a）。
- **需 PM 拍板**：**已拍板 C**；范围承接 (a)/(b) 待用户确认。

### R-3 · 本期不做 Provider 层抽象的代价

- **现状**：v0.5.1 做完后，`packages/provider/` 有两个具体 Provider（Qwen / DeepSeek）并列存在；若未来再接 OpenAI / Anthropic / Ollama，每家都重复一套适配会累积熵。
- **本期决策**：**接受此代价**。理由：两家 Provider 不足以验证抽象基类的合理形状，强行抽象容易"过早抽象、事后翻工"。
- **后续**：**接入第三家 Provider 前**必须先做一次 Provider 层抽象（推测在 OpenAI 或 Ollama 接入时触发），届时单独立需求。
- **需 PM 拍板**：**确认本期不做抽象**（对齐当前默认）。

### R-4 · DeepSeek API 流式分块的兼容性细节（低风险，developer 阶段）

- DeepSeek API 是"类 OpenAI"，但不等于 "完全一致"；`tool_calls` 在流式分块中的切分位置、`finish_reason` 枚举值、`usage` 何时返回等细节可能存在差异。
- **本期策略**：developer 在实装时按实际 API 行为写适配，不在本需求文档里做选型。
- **需 PM 拍板**：**无**。若 developer 发现为了兼容 DeepSeek 必须修改 `ChatChunk` 契约（目前判断不必），需回到 PM 这边单独走变更审批。

### R-5 · 新增"测试连接"按钮的范围争议（小概率）

- 当前配置页是否有"测试连接"按钮尚未在本文档核实。
- **本期策略**：**如果有**，DeepSeek 同样能用；**如果没有**，本期不新增（属 Out-of-Scope）。
- **需 PM 拍板**：**开发启动时 developer 检查一下即可**，无需 PM 单独决策。

### R-6 · CWS 审核对新增 Provider 的影响（低）

- `host_permissions: <all_urls>` 已在 v0.4.0 放开，新增一家 Provider 不触发新的 CWS 审核要求。
- **需 PM 拍板**：**无**。
